# 项目说明

HMTrans（HM互传）是面向 macOS 与 HarmonyOS MatePad 的开源局域网工具。稳定版已经提供双端原文件互传；`v0.3` 在不改变本地直连原则的前提下，新增 MatePad 到 Mac 的实时屏幕投放。

## 产品原则

- **纯本地**：设备发现、文件和画面均在局域网内直连，不提供云端中转、账号或远程服务器。
- **明确授权**：配对表示信任当前应用安装；`v0.3.0` 投屏只从 Pad 发起。计划中的 `v0.3.1` 即使允许 Mac 发送启动请求，每次仍必须在 Pad 的系统录屏界面明确授权。
- **质量优先**：文件传输保持原字节与 SHA-256 校验；投屏优先保证可读、低延迟和方向切换稳定。
- **用户可控**：文件任务、投屏会话、配对关系和本地数据都有清晰状态与停止入口。
- **空闲低耗电**：没有文件任务或投屏会话时，不申请后台长时任务，也不运行高频编码、解码或界面刷新。
- **轻量开源**：优先采用 HarmonyOS 与 macOS 原生框架，不引入云 SDK、广告 SDK 或账号体系。

## 版本状态

- 当前稳定版为 `v0.3.0`。macOS 通过 GitHub Release 分发；HarmonyOS 已完成目标设备测试，待使用应用市场正式 Profile/证书签名并上传审核。
- `v0.3.0` 已把 MatePad 到 Mac 的单向画面投放纳入稳定能力，同时继续维护 `v0.2.1` 的文件互传链路。
- `v0.3.0` 只实现 MatePad 到 Mac 的单向画面投放；声音、触控、键盘和鼠标反向控制属于后续 `v0.3.x`。
- `v0.3.1` 计划增加 Mac 主动请求 Pad 投屏，以及经过 Pad 用户单独授权的触控、键盘和鼠标反向操作；当前包不包含这两项能力。
- 正常覆盖升级保留应用安装身份、配对关系、设置和历史；清除应用数据或完整卸载后生成新身份，需要重新输入一次六位配对码。

## 当前产品形态

双端继续采用三个一级入口，不为投屏增加第四个主 Tab：

1. **连接**：发现、六位码配对、选择设备、发送文件或文件夹，并在 Pad 上向已连接 Mac 发起投屏。
2. **历史**：管理文件传输任务与历史。投屏是实时会话，不写入文件传输历史。
3. **设置**：管理发现、接收、后台保护、下载目录、临时空间、网络服务、诊断、隐私和投屏能力。

Mac 的投屏画面在独立观看窗口显示，可切换普通窗口、画中画和全屏；画中画使用小型置顶窗口，方便录制整个 Mac 屏幕时同时展示 HMTrans 主窗口与 Pad 操作。Pad 在系统录屏授权后可以离开 HMTrans，继续演示其他应用。停止投屏可从 Pad 状态入口或 Mac 观看窗口完成。

## 兼容与边界

- `v0.3` 保留 `v0.2.1` 的文件/文件夹、断点续传、多设备、后台任务、历史和本地数据库能力。
- 文件传输协议维持 `0.2.0` 命名空间；投屏使用独立协议、端口和状态机，不能把视频帧当作文件任务处理。
- `v0.3.0` 不采集声音、不把 Mac 当副屏、不内置录屏、不提供反向控制、不做公网或设备中继。
- 同一 Pad 只允许一个投屏会话；同一 Mac 最多接收两台 Pad，双路强制不高于 1080P 级 / 30P，第三路或超出资源预算的会话会被明确拒绝。文件传输仍可并存。
- 屏幕画面属于敏感内容，诊断与历史不得保存画面、帧数据、配对码或会话密钥。

## 系统基线

- HarmonyOS 6.1.0 / API 23+，MatePad，全屏、横竖屏、分屏与浮窗。
- macOS 26+，Swift 6、SwiftUI/AppKit、VideoToolbox、AVFoundation/CoreMedia 和系统材质。
- Mac 构建继续使用 ad-hoc 签名；项目没有 Developer ID 和 Apple notarization，不在文档中宣称已完成正式签名或公证。
