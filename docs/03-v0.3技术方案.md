# v0.3 技术方案

本文定义 `v0.3.0` MatePad 到 Mac 单向投屏的技术基线。稳定版文件互传链路继续使用现有实现；投屏必须使用独立模块、端口、协议和状态机，不能侵入文件任务语义。

## 1. 范围与约束

- 发送端仅为 HarmonyOS MatePad，接收端仅为 macOS。
- 首版只传 H.264 画面，不传声音，不接收 Mac 的键盘、鼠标或触控输入。
- 首版同一 Pad 只建立一个投屏会话，同一 Mac 只显示一个 Pad 会话。
- 设备必须先通过现有六位码完成配对并保持双向在线；每次投屏仍需 HarmonyOS 系统屏幕采集授权。
- 文件传输协议继续维持 `0.2.0`；投屏协议版本为 `0.3.0`，默认使用独立 TCP 端口 `51890`。
- 投屏不落盘、不进入文件历史、不复用文件 Staging，不上传到服务器。

## 2. 总体架构

```text
HarmonyOS MatePad                              macOS
┌──────────────────────────┐                  ┌──────────────────────────┐
│ ArkUI 连接页 / 会话状态   │                  │ SwiftUI 邀请与观看窗口    │
├──────────────────────────┤                  ├──────────────────────────┤
│ PadScreenCastController  │  控制消息/心跳    │ ScreenCastReceiverService│
│ ScreenCastSenderService  │ ───────────────▶ │ ScreenCastManager        │
├──────────────────────────┤  H.264 码流       ├──────────────────────────┤
│ NativeScreenCast N-API   │ ───────────────▶ │ VideoToolbox 解码         │
│ 屏幕采集 + 硬件编码       │                  │ CoreMedia / 原生图层显示  │
└──────────────────────────┘                  └──────────────────────────┘
```

### HarmonyOS 模块

- `PadScreenCastController`：维护会话状态、用户操作、设置持久化和页面可观察状态。
- `ScreenCastSenderService`：完成握手、加密二进制帧、心跳、背压、有限重连和停止通知。
- `screen_capture_module`：通过 N-API 暴露最小接口，内部使用 `OH_AVScreenCapture` 与 `OH_AVCodec` 完成系统屏幕采集和硬件 H.264 编码。
- `PadPreferencesService`：保存画质档位和投屏端口，不保存屏幕内容。

### macOS 模块

- `ScreenCastReceiverService`：监听投屏端口、限制连接、校验配对密钥、解析和解密帧。
- `ScreenCastManager`：维护会话生命周期、统计、错误映射和观看窗口入口。
- `ScreenCastVideoRenderer`：使用 `VTDecompressionSession` 与 `AVSampleBufferDisplayLayer` 解码显示 H.264。
- `ScreenCastWindowController`：管理独立窗口、画中画、全屏、位置恢复、睡眠和应用退出。

新增模块按职责拆分，投屏新增文件保持在 600 行以内；既有页面和控制器保持低于 1000 行，超过该阈值必须继续拆分。页面不得直接持有原生编码器或网络连接。

## 3. 会话状态机

```text
idle
  └─ 用户发起 ─▶ inviting
                    ├─ 对端拒绝/超时 ─▶ failed ─▶ idle
                    └─ 对端接受 ─▶ authorizing
                                      ├─ 用户拒绝 ─▶ stopped ─▶ idle
                                      └─ 系统允许 ─▶ starting
                                                        ├─ 首帧成功 ─▶ streaming
                                                        └─ 失败 ─▶ failed
streaming
  ├─ 网络暂断 ─▶ reconnecting ─┬─ 恢复并收到关键帧 ─▶ streaming
  │                            └─ 超时 ─▶ failed
  └─ 任一端停止/系统撤销 ─▶ stopping ─▶ stopped ─▶ idle
```

- 状态变更只能由 `PadScreenCastController` / `ScreenCastManager` 统一提交。
- 所有异步回调必须绑定 `sessionId`，过期会话的回调不得修改新会话。
- `stop` 必须幂等，并按采集器、编码器、发送队列、连接、解码器、显示层的顺序释放资源。

## 4. 采集与编码

HarmonyOS 端优先采用原生 surface 链路，减少屏幕帧在 JS、CPU 内存和 GPU 间的复制。正式实现前先完成原生能力验证，不得用高频截图轮询替代系统屏幕采集。

| 模式 | 目标分辨率 | 帧率 | 使用场景 |
| --- | --- | --- | --- |
| 自动 | 在 1080p 与 720p 间调整 | 最高 30 fps | 默认 |
| 流畅 | 最长边约 1280 | 30 fps | 网络或设备负载较高 |
| 清晰 | 最长边约 1920 | 30 fps | 演示文字与界面细节 |

编码要求：

- H.264 低延迟配置，不使用 B 帧。
- 启动、方向变化、重连和解码异常后主动生成 IDR。
- 关键帧间隔初始为 1–2 秒，后续由真机数据调整。
- SPS/PPS 在会话开始、编码参数变化和重连后重新发送。
- 时间戳使用单调时钟微秒值，不能使用会跳变的本地日期时间。
- 方向变化时先发送新视频配置，再发送对应分辨率的关键帧。

## 5. 传输与背压

`v0.3.0` 首版使用一条可靠 TCP 连接承载控制与 H.264 二进制帧，降低跨平台实现复杂度；文件传输继续使用原端口和连接。后续只有在真机数据证明 TCP 队头阻塞不可接受时，才评估 QUIC 或其他传输。

- 控制消息最大 64 KiB。
- 单帧载荷最大 8 MiB。
- Pad 待发送视频最多保留 4 帧或 32 MiB，以先达到的上限为准。
- 发生积压时丢弃最旧的非关键帧，保留最新关键帧边界，并通知编码器降低码率。
- Mac 解码队列最多保留 4 帧；解码器恢复后请求新关键帧，不连续播放过期画面追赶。
- 心跳周期 2 秒；连续 6 秒无心跳进入重连，重连总窗口 10 秒。
- 投屏与文件传输使用独立端口和有界队列；当前版本不动态改写文件任务并发。双链路并行时必须保持文件任务 ID、断点、SHA-256 校验和投屏会话状态彼此独立。

## 6. 解码与显示

- Mac 收到视频配置后创建或重建 `VTDecompressionSession`。
- 解码输出按 PTS 排序显示，不人为增加大缓存。
- 观看窗口默认适应内容并保持原始宽高比，补边不拉伸。
- 窗口缩放只改变显示层，不触发 Pad 编码分辨率频繁抖动。
- 普通窗口、画中画和全屏复用同一显示模型与解码会话；切换时只迁移显示层或渲染目标，不重新连接网络。
- 画中画使用无厚重标题栏的 AppKit 浮动窗口，保持比例、允许缩放、限制在可见屏幕范围，并把最后位置和大小保存在本地设置中。
- 方向变化、全屏、画中画和 Retina 缩放由显示层处理；统计信息使用低频刷新，不随每帧触发 SwiftUI 全树更新。

## 7. 身份、安全与隐私

- 只有现有配对身份匹配且保存了 256 位随机共享密钥的设备可以进入投屏握手；IP 地址、设备名称和六位码都不能代替身份密钥。
- 每次发起生成随机 `sessionId`；连接内序号严格单调递增，重复或乱序的已认证控制包直接拒绝。
- `HELLO` 只携带会话和设备标识，不携带密钥；Mac 用本地配对记录找到对应共享密钥，验证成功后才发送加密 ACK。
- 配置、视频、控制、心跳和结束载荷统一使用 AES-256-GCM；每包生成独立 96 位随机 nonce，固定头作为 AAD，密文自带完整性标签。六位码只用于首次人工确认，绝不作为媒体密钥。
- 日志与诊断不得保存帧、SPS/PPS 原值、配对码、会话密钥、完整 IP 或完整本地路径。
- Mac 只在内存中解码显示。用户使用系统录屏生成的文件由用户和 macOS 管理。

## 8. 生命周期与资源管理

### Pad

- 只有系统授权成功且会话启动后才申请数据传输后台能力。
- 系统停止录屏、应用被终止、锁屏策略不允许采集或编码器异常时立即结束会话。
- 页面隐藏不能销毁控制器；应用退出或 `aboutToDisappear` 必须清理定时器和监听器。
- 空闲时不保留采集 surface、编码器、发送线程或高频心跳。

### Mac

- 主窗口关闭不终止独立观看窗口。
- 睡眠、应用退出、接收服务关闭或用户停止时发送结束原因并释放解码器。
- 唤醒后只恢复发现和接收监听，不自动恢复屏幕采集。
- 连接、解码和显示资源都必须有上限，并在异常路径中释放。

## 9. 错误与诊断

错误统一映射为投屏错误码、用户文案和可恢复动作。诊断至少包含：

- 会话阶段和结束原因。
- 编码分辨率、帧率、目标/实际码率。
- 首帧耗时、往返延迟、发送队列、解码丢帧和重连次数。
- 平台版本和投屏协议版本。

用户界面不得直接显示原始异常对象。标准错误对象使用 `message`，未知对象使用安全字符串化并限制长度。

## 10. 测试策略

- 协议：帧拆包/粘包、长度上限、未知类型、保留位、版本协商、无效密钥和重复序号。
- 状态机：授权拒绝、邀请超时、启动取消、重连、双方同时停止和旧回调污染。
- 编解码：SPS/PPS、方向变化、关键帧丢失、解码器重建和异常 NALU。
- 性能：1080p/30、720p/30、60 分钟稳定运行、前后台切换和大文件并行。
- 安全：未配对设备、身份变化、认证失败、重复序号、超大帧和连接风暴。
- 回归：`v0.2.1` 文件/文件夹、断点续传、多设备、历史和设置全部继续通过。

## 11. 依赖与构建

- HarmonyOS：ArkTS/ArkUI、系统屏幕采集、原生 AVCodec、N-API 与必要的 Native C++ 构建配置。
- macOS：Swift 6、SwiftUI/AppKit、VideoToolbox、AVFoundation、CoreMedia、QuartzCore。
- 保持当前最低系统：HarmonyOS 6.1.0 / API 23+，macOS 26+。
- Mac 继续使用 ad-hoc 签名；没有 Developer ID 和 notarization，不把该限制包装成已完成能力。
