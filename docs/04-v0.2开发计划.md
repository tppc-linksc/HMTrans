# v0.2 开发计划

> 2026-07-12 状态：主要功能已进入发布候选代码；本文件保留工作包和依赖关系作为追溯基线。当前完成度、构建证据和发布阻断项以《06-v0.2发布前检查报告》为准，不能继续用原始人日估算判断是否已经验收。

## 1. 版本目标

v0.2 的交付目标是：纯局域网前提下，完成可后台、可控制、可恢复、可诊断的一对一和多设备原文件传输。

本计划记录 v0.2 原始工作包，不重复罗列 v0.1 已完成功能。当前市场版本的首次启动连接问题已在发布候选中实现修复，仍等待应用市场正式包验收。

## 2. 优先级

### P0：没有这些能力不能发布 v0.2

- 应用市场首次启动发现修复，隐私、目录、UDP 和 TCP 状态解耦。
- 应用自有隐私弹窗，同意前不启动局域网服务。
- 双端传输协调器、统一状态机和本地数据库。
- 用户不可见的持久临时文件和可靠最终发布。
- 暂停、继续、取消、重试、断点续传和进程重启恢复。
- 活动传输后台保障，切换页面、关闭窗口和短时锁屏不中断。
- 任务中心、基础人工控制、错误码和诊断信息。
- 空闲能耗治理，移除高频轮询和广播。

### P1：v0.2 完整功能

- 六位配对码和设备身份记录。
- 文件夹语义传输和自动安全解压。
- 一对多、群组任务和失败隔离。
- Mac 状态栏自绘图标、拖拽轻量发送面板和悬浮任务列表。
- 已配对设备管理、完整设备信息、手动发现/接收开关。
- 记录删除、清空和临时空间管理。
- HarmonyOS 后台通知控制。

### P2：完成度和视觉质量

- 文件类型图标和异步缩略图。
- 历史搜索、筛选和多选操作。
- 空状态、动画、无障碍和深浅色细节。
- 诊断导出和长时间运行可观测性完善。

P2 仍属于 v0.2 范围，但在传输状态机稳定后实现，避免 UI 反复返工。

## 3. 阶段与工作包

### 阶段 A：发行版稳定性和能耗止损

目标：先解决市场版互相搜不到和 Mac 空闲耗电偏高。

- 用应用自己的隐私弹窗替换新版本的托管隐私确认依赖。
- 隐私同意前不启动发现、接收和目录初始化。
- 下载目录、UDP 发现和 TCP 接收独立启动、失败和重试。
- HarmonyOS API 23+ 验证当前下载目录能力，移除无效的旧权限硬门槛。
- 根据活动网卡计算广播地址，已知设备使用单播补充。
- 移除 Mac 0.1 秒 RunLoop 唤醒、双端每秒设备轮询和 HarmonyOS 大量硬编码广播地址。
- 增加服务状态、手动刷新、重启发现、重启接收和基础诊断入口。

完成条件：应用市场干净安装可发现；空闲 30 分钟没有持续高频 CPU/网络唤醒。

### 阶段 B：架构、协议和存储基础

- 从 `TransferViewModel` 和 HarmonyOS 页面拆出 `TransferCoordinator`。
- 建立 v0.2 协议模型和 v0.1 兼容适配器。
- macOS SQLite3、HarmonyOS relationalStore 数据层。
- 建立任务、群组、设备、检查点、内部文件和诊断事件模型。
- 建立应用私有持久 Staging 目录和引用计数清理机制。
- 迁移现有信任设备设置，准备后续配对升级。

完成条件：双端可创建、持久化和恢复模拟任务；页面销毁不销毁协调器。

### 阶段 C：断点续传和任务控制

- 实现统一任务状态机。
- 接收端单一 `.part`、实际长度校正和完整分块回退。
- 发送端 seek 续传和最后分块哈希确认。
- 实现暂停、继续、取消、等待重连、失败继续和从头重传。
- 实现幂等 `transferId` 和最终名称预约。
- 检查点按时间/字节限频写入。
- 实现任务详情、错误说明和复制诊断信息。

完成条件：在 10%、50%、95% 主动断网和杀进程后恢复，不重头传、不生成重复文件。

### 阶段 D：后台生命周期

- HarmonyOS 声明并使用 `dataTransfer` 后台模式。
- 活动任务启动通知，完成后立即停止长时任务。
- 通知支持进度、暂停和取消。
- Mac 协调器独立于窗口，并在活动任务期间声明系统活动。
- 网络切换、Pad 锁屏、Mac 睡眠/唤醒后进入等待重连和自动恢复。
- 应用内部切换页面不重复注册后台任务、socket 或计时器。

完成条件：正常退后台和短时锁屏继续传；系统强制中断后重新打开能恢复状态和断点。

### 阶段 E：配对和设备管理

- 已完成过渡门禁：发现不等于连接；未输入并通过对端六位码校验时，不写入信任、不进入已连接列表，也不能发起文件传输。
- v0.1 由单边信任弹窗产生的旧信任不迁移，升级后双端需要重新完成一次六位码配对。
- 实现临时密钥交换、六位短认证码和用户双端确认。
- 随机安装标识和身份指纹进入应用本地持久化，不采集硬件序列号；诊断中不输出原值。
- 数据库保存设备指纹、名称、系统版本、能力和最后在线时间。
- 删除设备时清理信任关系和安全材料。
- 实现接收策略：已配对自动接收、每次询问、拒绝所有。
- 旧版设备显示基础兼容模式。

完成条件：身份变化必须重新配对，未配对设备不能被快捷拖拽静默发送。

### 阶段 F：文件夹语义

- 把文件夹压缩从系统临时目录迁到持久 Staging。
- 记录来源文件夹名、文件数、来源大小、载荷大小和归档类型。
- 使用快速压缩策略，已压缩文件降低压缩级别。
- 接收端校验后在私有目录解压，防止 ZIP 路径穿越。
- 最终只发布文件夹，历史不展示内部 ZIP。
- 任务取消、失败、过期时正确处理归档引用。

完成条件：中文、长文件名、隐藏文件、多层目录和恶意路径用例通过。

### 阶段 G：多设备和群组任务

- 设备列表支持多选和发送到全部。
- 一个父群组持有每台设备的独立子任务。
- 同一来源只准备和哈希一次。
- 双端接收器改为受控并发会话，不能共享可变进度状态。
- 并发上限初始为 Pad 2、Mac 3，根据测试调整。
- 群组支持全部暂停、全部继续、全部取消和只重试失败设备。

完成条件：至少两台接收设备并行，任一失败不影响其他任务完成。

### 阶段 H：Mac 状态栏快捷体验

- 绘制 HM互传专用状态栏模板图标并验证各种系统外观。
- 用 AppKit `NSStatusItem` 状态栏控制器替换现有简单 `MenuBarExtra`。
- 状态栏按钮注册文件 URL 拖放。
- 拖拽进入时打开锚定轻量 `NSPanel`。
- 一台已配对在线设备直接发送；多台设备明确选择或发送到全部。
- 无拖拽悬浮时展示最多 3 个活动任务的名称和进度。
- 精简面板提供暂停/继续、取消和打开任务中心。
- Finder、Dock、主窗口和状态栏拖拽共用统一任务入口。

完成条件：主窗口隐藏时可完整发起、查看和控制传输，面板关闭不影响任务。

### 阶段 I：UI、历史和设置完善

- 双端统一设备、任务、历史、设置的信息架构。
- 实现删除单条、多选删除、清除已完成、清除失败和清理临时数据。
- 增加完整错误页面、恢复入口和服务开关。
- 增加常见文件图标和低优先级缩略图。
- 适配 Mac 深浅色、高对比度以及 Pad 全屏、分屏和小窗。
- 所有状态支持键盘/辅助功能读取，不能只靠颜色。

完成条件：所有状态均有明确反馈和下一步操作，空状态不出现无说明空白页。

### 阶段 J：双端集成和正式包验收

- v0.2 ↔ v0.2 完整能力。
- v0.2 ↔ v0.1 基础一对一兼容。
- 调试包、正式签名包、应用市场包分别验收。
- 完成功能、性能、能耗、后台、异常、存储和长时间运行测试。
- 更新官网连接说明、隐私政策、下载页和发布说明。

完成条件：全部 P0/P1/P2 验收项关闭，未关闭问题有明确发布阻断判断。

## 4. 依赖关系

```text
A 发行稳定性
      │
      ▼
B 架构/协议/存储
      │
      ├── C 续传控制 ── D 后台
      ├── E 配对 ─────── H 状态栏快捷发送
      ├── F 文件夹 ───── G 多设备
      └───────────────── I UI/历史/设置
                              │
                              ▼
                         J 正式包验收
```

状态栏快捷发送不能先于统一任务入口和配对完成，否则会形成不可恢复、不可确认的第二套发送链路。

## 6. 开发纪律

- 新代码保留必要注释，状态机、续传、后台和配对等核心逻辑必须有详细注释。
- 双端协议字段先更新协议文档，再实现代码。
- 不以编译通过替代真机、后台、视觉和正式渠道包验证。
- 性能优化必须有前后测量，不用主观感受替代数据。
- 不引入云服务、账号 SDK 或重量级依赖扩大安装包。
- 每个阶段完成后同步本文状态和验收清单，不把未完成能力写成已交付。
