# 传输协议 v0.1

`v0.1` 使用一条 TCP 连接传输一个文件。文件夹会在发送端先压缩为 zip，再按单文件传输。

默认端口：

```text
51888
```

默认分片大小：

```text
1048576 bytes
```

## 消息格式

控制消息使用 JSON，并以换行符 `\\n` 结尾。

文件内容不是 JSON，而是在确认接收后直接发送原始二进制字节。

## 传输流程

1. 发送端连接接收端。
2. 发送端发送 `file_meta`。
3. 接收端根据设备信任状态决定是否弹出确认。
4. 接收端发送 `receive_decision`。
5. 如果同意，发送端发送原始文件字节。
6. 接收端读取 exactly `fileSize` 字节。
7. 接收端计算 SHA-256。
8. 接收端发送 `transfer_success` 或 `transfer_failed`。

## 文件元数据

```json
{
  "type": "file_meta",
  "app": "HMTrans",
  "version": "0.1.0",
  "transferId": "B96F52A5-8B6E-4B19-9D85-2C12F3F9E13C",
  "senderDeviceId": "mac-xxxxxx",
  "senderName": "MacBook Pro",
  "senderPlatform": "macOS",
  "fileName": "IMG_0001.DNG",
  "fileSize": 48234432,
  "sha256": "012345...",
  "chunkSize": 1048576,
  "totalChunks": 46
}
```

字段说明：

| 字段 | 说明 |
| --- | --- |
| `type` | 消息类型 |
| `app` | 固定为 `HMTrans` |
| `version` | 协议版本 |
| `transferId` | 本次传输 ID |
| `senderDeviceId` | 发送端设备 ID |
| `senderName` | 发送端展示名称 |
| `senderPlatform` | 发送端平台 |
| `fileName` | 文件名；文件夹会变成 zip 文件名 |
| `fileSize` | 文件字节数 |
| `sha256` | 原始文件 SHA-256 |
| `chunkSize` | 推荐分片大小 |
| `totalChunks` | 总分片数 |

## 接收决定

同意：

```json
{
  "type": "receive_decision",
  "accepted": true,
  "reason": null
}
```

拒绝：

```json
{
  "type": "receive_decision",
  "accepted": false,
  "reason": "user_rejected"
}
```

## 传输结果

成功：

```json
{
  "type": "transfer_success",
  "transferId": "B96F52A5-8B6E-4B19-9D85-2C12F3F9E13C",
  "sha256": "012345...",
  "reason": null
}
```

失败：

```json
{
  "type": "transfer_failed",
  "transferId": "B96F52A5-8B6E-4B19-9D85-2C12F3F9E13C",
  "sha256": "abcdef...",
  "reason": "hash_mismatch"
}
```

## v0.1 不包含的协议能力

以下能力留到 `v0.2`：

- 配对码字段与配对握手。
- 文件夹语义字段：原始类型、原始文件夹名、文件数量、归档载体类型、接收端自动解包规则。
- 断点续传所需的 offset、range、chunk hash。
- 暂停、继续、停止和后台恢复所需的 transfer state 消息。
- 后台传输保活、心跳、超时恢复和重新连接协商字段。
- 设备完整名称、平台、系统版本和能力字段。
- 一对多传输的 groupId、receiverId 和 per-device result。
- 失败文件的续传 token。
