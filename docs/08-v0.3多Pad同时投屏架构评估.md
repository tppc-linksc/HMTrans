# v0.3 多 Pad 同时投屏架构评估

## 1. 结论

多台 MatePad 同时向一台 Mac 投屏在技术上可实现：每台 Pad 独立完成屏幕采集和 H.264 编码，同一个 Mac TCP 监听端口可以同时接受多条连接，Mac 为每条已认证连接分别创建解密、解码、统计和显示资源即可。v0.3 当前代码已经按该模型开放最多两路独立投屏。

重构前实现有三处明确的单会话边界：

- `ScreenCastReceiverService.activeSession` 只允许一条已认证媒体会话。
- `ScreenCastManager` 只有一个状态、一个 `ScreenCastVideoRenderer` 和一组统计。
- `ScreenCastWindowRegistry` 只有一个窗口控制器。

本轮没有把 `activeSession` 简单改成数组，而是完成连接队列、密码器、解码器、统计和窗口的逐会话隔离。未经资源隔离的直接放开会导致不同 Pad 的 SPS/PPS、序号、解码器和窗口状态互相污染。

## 2. 数据流与会话模型

目标架构如下：

```text
Pad A ── HMCS/TCP ──┐
                    │   单一监听端口
Pad B ── HMCS/TCP ──┼─▶ Receiver ─▶ SessionRegistry
                    │                    ├─ Session A ─▶ Decoder A ─▶ Window/Tile A
Pad C ── HMCS/TCP ──┘                    ├─ Session B ─▶ Decoder B ─▶ Window/Tile B
                                         └─ Session C ─▶ Decoder C ─▶ Window/Tile C
```

一个 `ScreenCastSession` 必须独占以下状态：

- `sessionId`、`deviceId`、身份指纹和配对共享密钥。
- AES-GCM 密码器、收发序号、心跳、重连和帧解析器。
- H.264 SPS/PPS、关键帧边界和 `VTDecompressionSession`。
- 分辨率、帧率、码率、丢帧和错误状态。
- 自己的观看窗口或多屏网格中的一个 Tile。

同一投屏端口可以复用，不需要为每台 Pad 配置不同端口。TCP 监听器负责接受连接，`sessionId + deviceId` 负责区分会话。

## 3. Mac 端需要的重构

### 3.1 接收层

- 删除全局 `activeSession`，改为有容量上限的 `SessionRegistry`。
- 鉴权成功后按 `sessionId` 注册；同一设备重复会话需要拒绝或先显式停止旧会话。
- 每条连接使用独立串行队列或 Actor 解析、验序和解密，不能让高码率流在一条全局队列上互相阻塞。
- 回调必须携带 `sessionId`，过期会话不得更新其他设备状态。

### 3.2 解码与状态层

- `ScreenCastManager` 改为 `ScreenCastCoordinator + [SessionID: ScreenCastSessionModel]`。
- 每个会话拥有独立 `ScreenCastVideoRenderer` 和 `VTDecompressionSession`。
- 当前逐帧在 `MainActor` 同步解码的路径必须迁出主线程；主线程只接收低频状态和显示目标更新。
- VideoToolbox 创建会话时优先要求硬件解码；硬件资源不足时拒绝新会话，不能静默退回高占用软件解码。

### 3.3 显示层

- `ScreenCastWindowRegistry` 改为按 `sessionId` 保存多个窗口控制器。
- 支持两种显示方式：每台 Pad 独立窗口；一个总览窗口中的 2×1、2×2 网格。
- 关闭某个窗口只停止对应 Pad，会话之间不能联动停止。
- 左下角投屏入口显示当前会话数，并允许打开、停止或聚焦指定设备。

## 4. 资源准入与降级

多路是否可用不由“连接数量”单独决定，而由解码像素率、总码率、内存和温度共同决定。首个可验证版本建议：

- 默认最多两路，不直接承诺无限设备。
- 两路测试基线为 `1080P 级 / 30P`；通过持续真机测试后再开放两路 60P。
- 已有一路 2K/60P 时，新会话先要求降到 1080P/30P；用户不同意则返回资源繁忙。
- 文件传输与投屏并行时继续保持独立端口和队列，但总网络带宽需要纳入准入判断。
- Mac 解码帧率持续低于目标、显示队列积压或发生热降频时，只降低对应会话，不影响其他会话。

建议新增错误：

- `CAST-BUSY-001`：达到会话数量上限。
- `CAST-BUSY-002`：硬件解码器资源不足。
- `CAST-BUSY-003`：当前总像素率或网络预算不足，需要降低参数。

## 5. 安全与隔离

- 每个 Pad 继续使用各自的配对共享密钥和随机 `sessionId`；不同会话不能复用 nonce、序号或密码器。
- 一台设备认证、解码或网络失败只关闭该设备会话。
- 总览窗口只显示内存中的解码画面，不合成或保存录像。
- 诊断按匿名会话 ID 记录统计，不记录屏幕内容、密钥或完整局域网地址。

## 6. 当前实现与通过条件

1. 已把现有单会话拆成 `ScreenCastSessionModel`，每路拥有独立密码器、协议队列、解码队列、VideoToolbox 会话、统计和窗口。
2. 接收器最多接受两条已认证媒体会话；第三路、重复设备和不符合双路预算的参数会收到明确拒绝原因。
3. 双路强制使用不高于 `1080P 级 / 30P`；单路仍可按真机能力选择 2K/60P。
4. 已支持每台 Pad 独立窗口、独立停止、左下角入口显示当前路数；2×1/2×2 网格总览尚未加入。
5. 仍需使用两台真实 MatePad 完成 1080P/30P 双路至少 30 分钟测试，同时执行文件传输。
6. 只有在双路显示帧率、内存、CPU/GPU、温度和网络均达到验收线后，才考虑开放双路 60P。

如果没有两台真机或硬件解码资源测试不通过，发布说明必须将双路标记为测试功能，不得把编译通过当作真机性能验收。

## 7. 投屏网络测试

Pad 设置页提供“投屏网络测试”，复用投屏 TCP 端口和各自的配对共享密钥：

- 测试数据由 Pad 在内存中生成，不读取文件、不写临时文件、不访问公网。
- 首包完成设备身份校验，后续探测和数据包继续使用 HMCS AES-GCM 加密。
- 进行 10 次往返探测，计算平均 RTT 与相邻样本抖动。
- Pad 向 Mac 发送 32 MiB 固定载荷，Mac 按实际接收时间计算 Mbps。
- 测试时禁止已有投屏，避免压测流量干扰正在观看的会话。
- 结果按可用余量推荐 720P30、单路 1080P30、双路 1080P30、双路 1080P60或单路 2K60。
