# 开发计划

## 当前状态：v0.1 发版前调试

当前代码已经进入 `v0.1` 真机调试阶段。等 Mac 端和 MatePad 双向传输、连接恢复、历史记录、文件保存路径等问题确认稳定后，发布 `v0.1`。

已完成：

- Mac Swift 共享传输核心。
- Mac CLI 测试入口。
- Mac SwiftUI 图形入口。
- HarmonyOS DevEco / hvigor 工程。
- HarmonyOS ArkTS 文件选择、拖拽、TCP 发送接收、SHA-256 校验代码。
- 双端共用 JSON 行控制消息 + 原始二进制文件流协议。
- 双端启动后自动常驻接收。
- UDP 局域网发现和上次设备记忆。
- Mac 端拖拽发送、Dock 图标接收文件、菜单栏入口。
- 文件夹传输；发送端临时压缩为 zip。
- 当前传输和历史记录列表。
- 实时速度、文件大小、文件格式、平均速度展示。
- Mac / HarmonyOS 端 UI 已从原型表单改成沉浸卡片布局。

`v0.1` 发版前还要确认：

- Mac -> MatePad 传输后，MatePad 当前传输和历史记录显示正确。
- MatePad -> Mac 选择文件后能进入发送流程，不出现无反馈。
- 大文件传输稳定，速度和平均速度显示可信。
- 一端被杀进程后，另一端能较快清理旧连接并重连。
- 后台/锁屏期间的传输行为要明确验证：`v0.1` 记录限制，`v0.2` 做后台传输保证。
- MatePad 接收文件保存路径在 UI 中展示清楚。
- 真机安装流程、签名说明、README 开源说明完整。

## v0.1：首个开源版本

目标：

- 一对一传输可用。
- 不需要手动输入 IP，双端可以自动发现。
- 支持文件和文件夹。
- 支持拖拽和选择文件。
- 支持首次信任确认。
- 支持当前传输和历史记录。
- 支持速度、大小、格式、结果展示。
- 支持 SHA-256 完整性校验。

不做：

- 一对多传输。
- 群组发送。
- 配对码。
- 断点续传。
- 失败文件续传 UI。
- 公网传输。
- 端到端加密。

验收标准：

- Mac 能发现 MatePad。
- MatePad 能发现 Mac。
- 两端都能主动发送文件。
- 两端都能接收并确认文件。
- 文件和文件夹都能传输。
- 接收端历史记录能显示保存位置或失败原因。
- 常见大文件传输稳定。

## v0.2：多设备与失败恢复

`v0.2` 是下一阶段待做版本，重点解决多设备、断点续传和更可靠的安全确认。

目标：

- 一对多连接和群组发送。
- 文件夹语义传输：传输的是文件夹，接收结果也必须是文件夹；允许内部临时压缩，但用户不可感知。
- 断点续传，传输中支持暂停和停止。
- 已连接设备的传输记录恢复。
- 读取并展示连接设备的完整设备名称和系统版本。
- 配对码确认。
- 失败文件 UI 展示和续传入口。
- 常见文件类型图标和缩略图：按扩展名匹配图片、视频、文档、压缩包等图标，能安全读取时优先展示真实缩略图。
- 后台保活和后台传输保证：传输过程中退到后台或短时间锁屏不应中断；系统强制中断后可恢复。
- 更稳定的锁屏恢复和离线重连。

### 一对多连接和群组发送

- 同时维护多个可信设备的在线状态。
- 支持从设备列表中多选接收方。
- 支持创建临时传输群组，例如 Mac 同时发给多台 MatePad。
- 同一源文件只准备一次，按接收方分别建立 TCP 传输。
- 每个接收方独立显示进度、实时速度、平均速度和成功/失败状态。
- 群发失败时只重试失败设备，不重复发送给已完成设备。


### 文件夹语义传输

- `v0.1` 允许把文件夹临时压缩为 zip 传输，接收端可能看到 zip 包。
- `v0.2` 必须做到“怎么传就怎么收”：用户发送的是文件夹，接收端最终也显示和保存为文件夹。
- 内部仍可使用 zip / tar / 自定义归档作为传输载体，但归档、解包、临时文件清理对用户不可见。
- 当前传输和历史记录中展示原始文件夹名称、目录大小、文件数量、传输方向和平均速度，不展示内部归档文件名。
- 断点续传需要记录文件夹归档清单或分块状态，避免中断后只能重新打包整目录。

### 断点续传与记录恢复

- 连接过的设备再次连接时，保留并恢复与该设备相关的传输记录。
- 传输记录与断点续传状态绑定，重新连接后可以继续查看上次失败、未完成或可续传的文件。
- 传输中记录支持暂停和停止；暂停后保留当前 offset，停止后转入失败/可重试记录。
- 删除可信设备时，后续需要提供是否同时清除该设备传输记录的选择。
- 失败记录保存 transferId、文件名、文件大小、SHA-256、已完成字节数、目标设备。
- 接收端保留 `.part` 临时文件和已写入大小。
- 重新连接后发送端先询问接收端已有 offset。
- 从 offset 继续发送剩余字节。
- 完成后仍按完整文件 SHA-256 校验。

### 设备信息展示

- 发现和连接消息需要携带完整设备名称、平台、系统版本和可用能力。
- Mac 端显示 MatePad 的完整设备名和 HarmonyOS 版本。
- MatePad 端显示 Mac 的电脑名称和 macOS 版本。
- 设备名称过长时 UI 可以截断，但详情或设备卡片展开态必须能查看完整名称。

### 后台保活与后台传输

- 传输过程中 App 退到后台、窗口关闭或短时间锁屏，不应导致正在传输的文件直接失败。
- Mac 端菜单栏常驻和接收服务常驻，窗口关闭不影响发送、接收和进度更新。
- HarmonyOS 端使用 HarmonyOS 6.1/API 23+ 后台任务能力，尽量保持传输 socket 和接收服务。
- 如果系统强制暂停或杀进程，记录 transferId、offset、临时文件路径和校验状态，重新连接后进入断点续传。
- UI 需要显示后台传输状态、系统限制提示、恢复状态和失败原因。

### 配对码

- `v0.1` 不做配对码。
- `v0.2` 首次连接使用短配对码确认，降低误信任陌生设备的风险。
- 配对成功后保存设备 ID 和信任关系。
- 删除设备后重新连接需要重新配对。

### 文件图标和缩略图

- 为常见文件类型准备统一图标，包括图片、视频、PDF、Office 文档、压缩包、文件夹和未知文件。
- 图片和视频在权限允许且性能可控时优先展示真实缩略图。
- 缩略图生成失败时回退到扩展名图标，不影响传输记录展示。
- Mac 端和 HarmonyOS 端使用同一套视觉语义，避免两端图标风格不一致。

### 失败文件 UI 与历史记录管理

- 当前传输失败后不只进入普通历史记录，还要进入“可重试/可续传”状态。
- 失败项展示失败原因、已传大小、目标设备、最后失败时间。
- 如果满足续传条件，展示“继续传输”。
- 如果不能续传，展示“重新发送”。
- 一对多场景下，每个接收方单独显示失败和续传入口。
- 历史记录支持删除单条记录和清空全部记录；清空只影响 UI 记录，不删除已保存文件，除非用户明确选择同时删除文件。

验收标准：

- 一台发送端可以同时向两台及以上接收端发送文件。
- 任一接收端失败不会影响其他接收端完成。
- 大文件中断后可以从已完成位置继续。
- 传输中可以暂停和停止，暂停后可以继续，停止后可以从失败记录重试或续传。
- 失败文件在 UI 上清楚可见，并能继续或重新发送。
- 历史记录可以删除单条或清空全部。
- 后台传输行为可验证：退到后台或短时间锁屏后仍能继续传输；被系统强制中断后能恢复或续传。
- 双端设备卡片能展示完整设备名称和系统版本。
- 首次连接需要配对码确认。
