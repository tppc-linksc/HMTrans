# v0.2 技术方案

## 1. 范围与约束

v0.2 是纯局域网传输引擎升级，不引入云端、账号或服务器依赖。

```text
发现：局域网 UDP
身份：本地配对和系统安全存储
传输：局域网 TCP
状态：本机数据库
断点：本机持久文件
校验：本机 SHA-256
```

平台基线：HarmonyOS 6.1.0 / API 23+，macOS 26+。

## 2. 总体架构

当前双端页面层承担了发现、接收、发送和历史状态。v0.2 必须先拆分生命周期，避免切换页面、关闭窗口或弹出状态栏面板时重复创建服务。

```text
UI
├── Mac 主窗口 / 状态栏面板
└── HarmonyOS 设备 / 任务 / 历史 / 设置
          │
          ▼
TransferCoordinator
├── DeviceSessionManager   发现、在线状态、配对、能力协商
├── TransferEngine         队列、发送、接收、暂停、续传、取消
├── TransferStore          任务、群组、检查点、历史
├── StagingStore           私有临时文件、文件夹归档、空间清理
├── BackgroundController   后台任务、锁屏、唤醒和恢复
├── DiagnosticsStore       结构化事件、错误码和导出
└── ProtocolAdapter
    ├── v0.1 基础兼容
    └── v0.2 完整协议
```

所有入口，包括 Finder 拖到主窗口、Dock、状态栏图标以及 Pad 文件选择器，都只负责创建统一任务，不单独实现发送流程。

## 3. 本地数据设计

### 3.1 选型

- macOS：系统 SQLite3，薄封装，不引入外部 ORM。
- HarmonyOS：系统 `relationalStore`。
- 安装身份：双端生成随机安装标识和随机身份指纹，保存在应用本地偏好；当前版本不采集硬件序列号，也不把该随机值描述为密码学私钥。
- 文件、分片、ZIP 和缩略图只保存路径，不写入数据库。

数据库使用事务、索引和批量更新。数据库写入在独立串行队列执行，不能阻塞网络循环和主线程。

### 3.2 逻辑表

`devices`：稳定设备 ID、名称、平台、系统版本、协议范围、能力、最后地址、最后在线时间、配对时间、身份指纹和用户备注。

`transfer_groups`：群组 ID、来源名称、来源类型、设备数量、整体状态、创建和完成时间。

`transfers`：传输 ID、群组 ID、方向、对端、来源和载荷类型、显示名称、来源路径、Staging 路径、最终路径、大小、哈希、状态、进度、错误和时间。

`checkpoints`：传输 ID、确认偏移、分块大小、最后完整分块哈希、更新时间和恢复代次。

`artifacts`：内部文件 ID、路径、用途、大小、引用数、创建时间、最后使用时间和清理期限。

`diagnostic_events`：时间、级别、模块、任务 ID、设备 ID、错误码和脱敏消息。

### 3.3 任务状态

```text
queued
preparing
pairing
awaiting_accept
transferring
paused
waiting_reconnect
verifying
extracting
completed
failed
cancelled
```

状态变化必须由状态机校验并写入数据库。应用重启时只恢复可恢复状态；`transferring` 在进程重启后先转换为 `waiting_reconnect`，不能假装仍在传输。

## 4. 私有临时文件与最终发布

未完成文件不能直接出现在公开下载目录，也不能放入系统可能自动清理的 `cacheDir` 或临时目录。

推荐目录：

```text
macOS Application Support/HMTrans/TransferData/
HarmonyOS filesDir/HMTrans/TransferData/
├── incoming/{transferId}/payload.part
├── outgoing/{artifactId}/payload
├── folders/{artifactId}/archive.zip
└── thumbnails/{artifactId}
```

接收完成流程：

1. 在私有持久目录接收唯一的 `payload.part`。
2. 校验完整 SHA-256。
3. 文件夹在私有目录解压，并防止绝对路径和 `../` 路径穿越。
4. 生成无冲突的最终名称。
5. 优先通过同卷移动/原子重命名发布到 `Download/HMTrans`。
6. 发布成功后更新数据库并清理内部文件。

用户默认只看到成功文件。若特定 HarmonyOS 版本不支持从私有目录原子移动超大文件，先做 10GB/50GB 技术验证；兜底只能使用一个隐藏临时对象，禁止产生多个用户可见分片。

删除规则：

- 删除已完成记录不删除用户文件。
- 删除未完成任务时明确询问是否同时删除可恢复数据。
- 群发任务共享的归档通过引用数管理，所有子任务结束或取消后才能清理。
- 默认保留未完成数据 7 天；自动清理前排除活动、暂停和等待重连任务。
- 低磁盘时提示用户处理，不静默删除仍可恢复的任务。

## 5. 发现和连接

设备发现、下载目录和 TCP 接收器是三个独立状态机。目录不可用时仍可发现设备和发送文件；接收器失败不能销毁发现服务。

发现策略：

- 从活动网卡和子网掩码计算广播地址，不维护大量硬编码网段。
- 前台打开设备页或手动刷新时使用短时快速广播。
- 找到设备后降低频率；已配对设备优先单播。
- 网络路径变化时立即重新广播并刷新地址。
- 空闲后台不启动传输长时任务，不进行每秒轮询。
- TTL 清理使用低频容差计时器或事件调度，不在主线程每秒扫描。

连接前通过 `hello/hello_ack` 协商协议范围和能力。v0.2 双端启用完整功能；任一端只支持 v0.1 时回退到一对一完整文件传输。

## 6. 断点续传

默认分块为 1MiB。TCP 仍负责可靠有序字节流，分块用于检查点和恢复，不为每个块创建独立文件。

恢复流程：

1. 发送端以原 `transferId` 重新发送 `transfer_offer`。
2. 接收端检查数据库和 `payload.part` 实际长度。
3. 实际长度回退到最后一个完整分块边界。
4. 接收端返回 `resumeOffset` 和最后一个完整分块哈希。
5. 发送端验证后 seek 到偏移继续发送。
6. 完成后执行全文件 SHA-256；不一致则进入可诊断失败，不直接覆盖最终文件。

检查点策略：每 4～8MiB 或每 1 秒保存一次；用户暂停、连接断开、进入后台和系统准备挂起时立即保存。数据库偏移仅作参考，恢复时以磁盘实际内容为准。

相同 `transferId` 必须复用已预约的最终名称和 `.part`，不能产生 `文件-1` 等重复任务。

## 7. 文件夹语义和多设备

协议区分 `sourceKind=folder` 与 `payloadKind=zip`。UI、历史和最终文件使用原始文件夹名称，内部 ZIP 不暴露。

- 使用 UTF-8 ZIP，避免携带不必要的 macOS resource fork 元数据。
- 文件夹准备结果保存到持久 Staging，应用重启后可继续使用。
- 解压先进入私有临时目录，校验路径后整体发布。
- 同一来源只准备和计算一次哈希。

群发使用父群组加独立子任务。每个接收方单独连接、进度、暂停、重试和结果，单台失败不影响其他设备。默认并发建议 Pad 2、Mac 3，最终通过性能测试调整。

## 8. 配对和本地安全

首次连接由发起端输入接收端界面显示的 180 秒六位短认证码。接收端验证短码后，双方保存对方随机安装标识和身份指纹；以后只有标识与指纹同时匹配才视为已配对。

- 数据库保存公开身份、指纹和信任状态。
- 私钥进入系统安全存储。
- 删除已配对设备时同步删除数据库关系和对应安全材料。
- 身份变化或签名验证失败时强制重新配对。
- v0.2 的六位码用于同一局域网内的人工配对门禁和后续身份一致性检查，不是端到端加密或抗中间人协议；产品不声称文件内容已加密。
- 配对成功响应同时写入双方在线设备集合和本地设备快照；设备快照包含显示名称、平台、系统版本、最近地址、传输端口和最后在线时间。
- UDP 发现采用广播加限频单播回应：任一端收到对方广播后向来源地址直接回发自身信标，避免出现单向发现；单播回应按设备限频，防止双端相互回应形成报文环路。

## 9. 后台和生命周期

### HarmonyOS

- 模块声明 `dataTransfer` 后台模式。
- 只有活动传输时启动长时任务并显示系统通知，队列清空后立即停止。
- 通知展示名称、进度、速度、暂停和取消。
- 处理后台取消、低速、网络切换和系统挂起回调；挂起前保存检查点。
- 应用被杀后重新打开，从数据库和私有 `.part` 恢复为等待重连。
- 不使用活动传输长时任务维持全天候空闲发现。

### macOS

- 接收器和协调器独立于窗口生命周期，由应用级对象持有。
- 关闭主窗口和状态栏面板不影响任务。
- 活动传输期间使用系统活动声明，完成后立即结束。
- Mac 睡眠导致连接中断时保存检查点，唤醒和网络恢复后重新发现并续传。

后台保证的含义是：正常切后台和短时锁屏时尽量持续；系统强杀、深度休眠或网络丢失时不丢状态，并能恢复。不得承诺操作系统无法保证的永久存活。

## 10. Mac 状态栏技术实现

当前 SwiftUI `MenuBarExtra` 改为 AppKit 状态栏控制器：

- `NSStatusItem` 持有自绘模板图标。
- 自定义状态栏按钮视图注册 `.fileURL` 拖放类型。
- `draggingEntered` 打开锚定在图标下方的轻量 `NSPanel`。
- `performDragOperation` 解析 URL 并交给 `TransferCoordinator`。
- `NSTrackingArea` 处理无拖拽时的悬浮，延迟打开任务面板。
- 面板使用 SwiftUI 内容承载，但窗口、锚定、拖放和鼠标跟踪由 AppKit 管理。
- 主窗口、状态栏和 Dock 打开文件共用同一模型和队列。

不扩大透明拖放区域到其他状态栏图标，热区只覆盖 HM互传图标和已打开的轻量面板。

## 11. 隐私流程

v0.2 使用应用自己的隐私弹窗和自有隐私政策页面，不依赖托管隐私确认服务。

启动顺序：

```text
加载本地隐私版本
├── 未同意/版本变化：展示自有弹窗
└── 已同意：进入服务初始化
        ├── 加载数据库和设置
        ├── 启动发现
        ├── 检查接收目录
        └── 按状态启动 TCP 接收器
```

同意前不创建 UDP/TCP socket，不广播设备信息，不扫描局域网。系统权限仍由系统弹窗申请，并且只在对应功能真正需要时触发。

## 12. 性能与能耗

当前实现存在空闲高频行为：Mac 广播约每 2 秒一次并以 0.1 秒 RunLoop 间隔等待，双端设备清理约每秒一次，HarmonyOS 每 5 秒向大量硬编码地址广播。v0.2 必须移除这些空闲高频唤醒。

优化原则：

- 发现采用前台突发、网络变化触发和后台降频。
- 被动 TCP listener 不配套高频轮询。
- 使用带 tolerance/leeway 的系统计时器，不用短周期 RunLoop 自旋。
- UI 进度约每秒 4～5 次，后台通知约每秒一次。
- 数据库检查点每秒或每 4～8MiB，使用事务批量提交。
- 网络流使用固定缓冲区和背压，不整文件载入内存。
- SHA-256 边传边计算；恢复后保持校验一致性。
- 文件夹采用快速压缩；对已经压缩的媒体和归档降低压缩级别。
- 缩略图低优先级异步生成，活动传输时让路。
- 日志设置大小和保留期限，禁止无限写盘。

## 13. 错误和诊断

错误分为发现、配对、目录、连接、传输、校验、解压、后台、存储和协议十类。每个错误包含稳定错误码、用户说明、恢复建议和脱敏技术详情。

诊断导出包含应用/系统/协议版本、设备类型、任务 ID、端口状态、最近结构化事件和错误码，不包含文件内容、六位短码、身份指纹原值或完整敏感路径。

## 14. 主要风险

- HarmonyOS 私有目录到公共下载目录的大文件移动能力需要真机验证。
- Picker URI 在进程重启后的持续访问能力不一致，必要时发送前复制到 Staging。
- HarmonyOS 后台能力受系统策略约束，必须用正式市场包验证。
- 多设备并发可能挤占单设备带宽，需要动态并发和背压。
- v0.1/v0.2 错峰升级需要协议协商和降级测试。
