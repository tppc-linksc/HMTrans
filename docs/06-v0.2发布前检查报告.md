# HMTrans v0.2 发布前检查报告

更新日期：2026-07-12

本报告区分“代码/构建已验证”“基础真机已验证”和“仍需正式包或压力实测”。编译成功不能替代真机、后台、视觉与应用市场安装验收。

## 1. 当前结论

- 双端 v0.2 代码可构建，基础 Mac ↔ MatePad 文件互传已经由开发机实测通过。
- 发现、接收和下载目录已拆成独立状态：下载目录未授权不再关闭 UDP 发现，设备仍可被发现并向外发送。
- 发布前仍有阻断性实测项：应用市场干净安装授权、多 GB 大文件、锁屏/后台、三台真机拓扑、横竖屏/浮窗/深色真机视觉和能耗。
- 在上述项目完成并记录证据前，v0.2 只能称为“发布候选”，不能对外宣称正式验收完成。

## 3. 权限与隐私

- 应用使用自己的隐私确认弹窗；同意前不创建 UDP/TCP socket、不广播设备信息。
- HarmonyOS 下载目录权限在隐私同意后由系统弹窗申请。拒绝后接收不可用，但发现与发送保持可用；“下载目录 → 检查”可再次申请并验证 `Download/HMTrans`。
- 发送端通过系统文件选择器获得用户主动选择的 URI；选择后立即复制到应用私有持久 Staging，避免 Picker 临时授权在大文件或重启后失效。
- `KEEP_BACKGROUND_RUNNING` 只用于存在活动传输且用户开启后台保护的时间段。
- 隐私政策地址统一为 `https://hmt.tppc.top/privacy.html`，不再使用托管隐私确认页面。

## 4. 文件夹、多设备和拓扑正确性

### 文件夹与嵌套文件夹

- Mac 使用 `ditto` 递归归档，Pad 使用系统 zlib 目录压缩。
- 协议传输 `sourceKind=folder`、`payloadKind=zip`、原文件夹名、原大小和文件数。
- 接收端先校验 SHA-256，再在应用私有目录解压并发布同名文件夹；Mac 端会在解压前逐项拒绝绝对路径和 `..`，Pad 端使用系统目录解压能力并清理最终名称。公共下载目录不展示内部 zip 或多个分片。
- 仍需用空文件夹、深层嵌套、同名文件、中文/Emoji 名称、符号链接、路径穿越样本和大量小文件做 Pad 真机验收。

### 一对多与不同文件

- 同一次发送会对每个“文件 × 目标设备”创建独立任务 ID、进度、结果和恢复点。
- 文件或文件夹的准备产物可在同一发送组内共享；任一目标失败不会把其他目标标记失败。
- 并发上限：Mac 为 3 个发送作业，Pad 为 2 个；超过上限排队，避免同时压满 CPU、内存和磁盘。
- 可以在第一批仍传输时修改选中设备并发起第二批，因此支持不同文件发给不同设备；已经创建的任务不会因之后修改选择而改变目标。

### A-B、B-C 已连接而 A-C 未连接

- HMTrans 不实现局域网中继。A 只能直接发给已与 A 双向确认的 B；C 不会因为连接了 B 就在 A 上显示为已连接。
- B 可以独立维护与 A、C 的连接并分别收发，但不会自动把 A 的文件转发给 C。
- 此拓扑的三台真机并发、断连和身份去重仍需发布前实测。

## 5. 窗口、主题和原生材质

- HarmonyOS Ability 声明全屏、分屏和浮窗，最小窗口为 360 × 480；页面宽度小于 760 时切换紧凑布局，滚动区域从顶部排布。
- Pad 启动窗口为纯白背景和透明占位资源，不展示应用图标。
- Pad 提供深色资源并避免固定状态栏文字颜色；关键页面、卡片、分隔线和设备状态表面随系统主题切换。
- Pad Dock 使用 `@kit.UIDesignKit` 的原生 `HdsTabs`，材质为 `MaterialType.IMMERSIVE`、等级为 `MaterialLevel.EXQUISITE`，不是手绘彩色背景。
- 横屏基础布局已检查；竖屏、浮窗、分屏和深色模式仍需在目标 MatePad 上截图验收。

## 6. 构建与代码质量

- HarmonyOS：使用 HarmonyOS 6.1 / API 23 HMS SDK 完整构建通过；当前签名 HAP 约 808 KB。
- macOS：使用 macOS 26+ SwiftUI/AppKit 构建通过，应用主二进制约 2.4 MB，`codesign --verify --deep --strict` 通过。
- Mac Swift Package 测试 2 项通过，覆盖断点状态和嵌套文件夹接收发布；`Info.plist` 校验和 `git diff --check` 通过。
- 复杂协议、私有 Staging、共享文件夹产物、后台任务引用计数和身份去重已有说明性注释。
- Mac 根 ViewModel 已拆分到 972 行；`HMTransCore.swift` 981 行，`V02RootView.swift` 733 行。
- HarmonyOS `Index.ets` 仍有 1592 行，是当前明确的代码质量债务。页面、服务和持久化已经分离，但根页面的状态协调与回调还应继续拆成控制器，禁止继续堆入新功能。
- HarmonyOS 构建仍有部分 ArkTS “Function may throw” 和能力提示警告，集中在 TCP、偏好与持久化服务；不影响本次构建，但发布前应继续收紧异常边界并复核目标设备能力。
- 当前检查只能证明源代码和本机构建产物；本轮没有可用 HDC 目标，因此不能替代最终 HAP 的安装、启动和真机视觉证据。
