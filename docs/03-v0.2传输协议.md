# 传输协议 v0.2

本文描述当前双端代码实际使用的 `0.2.0` 线协议。它没有额外的 `hello/hello_ack` 或 `offer` 阶段；发现、配对和文件传输各自使用独立消息。

## 1. 常量与编码

| 项 | 值 |
| --- | --- |
| 应用标识 | `HMTrans` |
| 协议版本 | `0.2.0` |
| UDP 发现端口 | 默认 `51889`，用户可改 |
| TCP 传输端口 | 默认 `51888`，用户可改 |
| 分块大小 | `1048576` bytes |
| 控制帧 | UTF-8 JSON + `\n`，单行最多 64 KiB |
| 文件载荷 | 控制帧之后的原始 TCP 字节 |
| 完整性 | SHA-256 小写十六进制 |

接收端只接受 `app=HMTrans` 且 `version=0.2.0` 的配对/文件消息。当前不实现 v0.1 自动降级；跨版本升级时应更新两端并重新配对。

## 2. UDP 发现

发现包是一个 `DeviceInfo` JSON 数据报：

```json
{
  "app": "HMTrans",
  "version": "0.2.0",
  "deviceName": "Linksc的MatePad Pro",
  "platform": "HarmonyOS",
  "ip": "0.0.0.0",
  "port": 51888,
  "deviceId": "pad-install-id",
  "systemVersion": "HarmonyOS6.1.0.117 SP6",
  "networkName": "HUAWEI Q6",
  "identityFingerprint": "random-install-fingerprint",
  "acknowledgedDeviceId": "optional-peer-id"
}
```

- 广播包可以没有 `acknowledgedDeviceId`。
- 收到对端后，客户端向对端地址发送带对端 ID 的定向确认包。
- `ip` 是提示值；实现会根据实际数据报来源和地址可用性修正，防止 VPN/多网卡导致单向连接。
- 设备 ID 是随机安装标识，不是硬件序列号。

## 3. 六位码配对

发起方建立短 TCP 连接并发送：

```json
{
  "type": "pairing_request",
  "app": "HMTrans",
  "version": "0.2.0",
  "requesterDeviceId": "mac-install-id",
  "requesterName": "Linksc的MacBook Pro",
  "requesterPlatform": "macOS",
  "requesterSystemVersion": "macOS 27.0 Beta (26A5378j)",
  "requesterIP": "192.168.3.17",
  "requesterPort": 51888,
  "code": "246810",
  "requesterFingerprint": "random-install-fingerprint"
}
```

接收端返回一行后关闭连接：

```json
{
  "type": "pairing_response",
  "accepted": true,
  "reason": null
}
```

拒绝原因至少包括：

- `invalid_pairing_code`：码错误或过期。
- `protocol_incompatible`：应用/版本/身份字段不兼容。

配对成功只改变双方本地信任数据，不建立永久 TCP 会话。

## 4. 文件元数据

每个任务建立一个 TCP 连接，发送方先发送一行：

```json
{
  "type": "file_meta",
  "app": "HMTrans",
  "version": "0.2.0",
  "transferId": "stable-task-id",
  "senderDeviceId": "mac-install-id",
  "senderName": "Linksc的MacBook Pro",
  "senderPlatform": "macOS",
  "fileName": "照片.jpg",
  "fileSize": 7340032,
  "sha256": "64-char-lowercase-hex",
  "chunkSize": 1048576,
  "totalChunks": 7,
  "resumeSupported": true,
  "sourceKind": "file",
  "payloadKind": "file",
  "sourceName": "照片.jpg",
  "sourceSize": 7340032,
  "sourceFileCount": 1,
  "senderFingerprint": "random-install-fingerprint"
}
```

文件夹传输时：

- `sourceKind=folder`
- `payloadKind=zip`
- `fileName` 是内部 ZIP 载荷名
- `sourceName` 是用户看到的原文件夹名
- `sourceSize/sourceFileCount` 是归档前统计

接收端校验类型、应用、协议版本、任务 ID、非负安全整数大小、分块、SHA-256 长度和发送端信任后再接受。

## 5. 接收决策与断点

接收端返回：

```json
{
  "type": "receive_decision",
  "accepted": true,
  "reason": null,
  "resumeOffset": 2097152
}
```

`resumeOffset` 是接收端已经持久化的连续字节数。复用断点必须同时匹配 `transferId`、文件名、总大小和 SHA-256，且偏移位于 `[0,fileSize]`。不匹配时从 0 开始并清理旧分片。

常见拒绝原因：

- `user_rejected`
- `receiver_paused`
- `receiver_cancelled`
- `invalid_meta`
- `protocol_incompatible`
- `insufficient_space`

## 6. 载荷与结果

收到 `accepted=true` 后，发送端从 `resumeOffset` 开始写原始字节，不再插入 JSON 分块头。接收端读取到 `fileSize` 后关闭临时文件并计算 SHA-256。

成功：

```json
{
  "type": "transfer_success",
  "transferId": "stable-task-id",
  "sha256": "64-char-lowercase-hex",
  "reason": null
}
```

失败：

```json
{
  "type": "transfer_failed",
  "transferId": "stable-task-id",
  "sha256": null,
  "reason": "hash_mismatch"
}
```

发送端只有在收到匹配任务 ID 的成功结果后才标记完成。

实现层对控制等待设置上限：Pad 等待配对回应和 `receive_decision` 最长 30 秒，等待接收端完成持久化与 SHA-256 校验最长 10 分钟；Mac 默认操作读写等待为 30 秒。超时是可诊断的连接失败，不允许任务无限挂起。

零字节文件的 `fileSize=0`、`totalChunks=0`；接收端返回接受决策后不等待载荷字节，直接校验空文件哈希并返回结果。

## 7. 暂停、取消和重连

- 暂停发送：本地 `TransferControl` 在检查点阻塞；接收暂停会关闭当前连接并保留 `.part`。
- 取消保留：关闭连接、任务进入已取消，保留恢复资料。
- 取消删除：额外删除该任务分片、元数据和不再被其他子任务引用的私有源。
- 网络断开：任务进入等待重连；同一 `transferId` 重新建立连接并使用接收端返回偏移。
- 多设备发送：同一 `groupId` 下每个目标都有独立 `transferId`、TCP 连接、断点和最终结果。

## 8. 文件夹安全

ZIP 在应用私有目录完成接收和完整哈希校验后，还必须验证：

- 中央目录可读且大小、条目数在上限内；支持安全范围内的 ZIP64 计数。
- 中央目录范围没有越过归档边界，单条目、总展开大小和压缩比没有超过安全上限。
- 条目不是绝对路径、Windows 盘符路径或包含 `..`。
- 条目不是符号链接。
- 解压前再次检查私有 Staging 的可用空间并保留 64 MiB 安全余量。
- 解压只发生在私有临时目录；全部成功后才把原文件夹移动到下载目录。

## 9. 安全边界

协议用于可信本地网络中的直接传输。配对码可避免无意连接和普通陌生设备直接发送，但当前 TCP 内容是明文，没有 TLS/E2E，不应描述为加密传输。诊断信息不得记录六位码、身份指纹原值或文件内容。
