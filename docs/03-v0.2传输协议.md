# 传输协议 v0.2

本文定义 HMTrans `0.2.0` 目标协议。实现期间如字段变化，必须先更新本文，再同步双端代码和兼容测试。

## 1. 基本约定

| 项目 | 值 |
| --- | --- |
| 发现端口 | UDP `51889` |
| 传输端口 | TCP `51888` |
| 控制消息 | UTF-8 JSON，以 `\n` 结束 |
| 文件载荷 | TCP 原始二进制流 |
| 默认分块 | `1048576` bytes |
| 完整性校验 | SHA-256，小写十六进制 |
| 协议版本 | `0.2.0` |

一个 TCP 数据连接只承载一个接收方的一个子任务。群发由多个独立连接组成，不在一条连接中复用多个设备或文件。

## 2. 通用消息头

每条 JSON 控制消息包含：

```json
{
  "type": "hello",
  "app": "HMTrans",
  "protocolVersion": "0.2.0",
  "messageId": "7E424742-1F2C-4A43-8D17-FF0EED8C126B",
  "timestamp": 1783700000000
}
```

| 字段 | 说明 |
| --- | --- |
| `type` | 消息类型 |
| `app` | 固定为 `HMTrans` |
| `protocolVersion` | 当前消息协议版本 |
| `messageId` | 消息唯一 ID，用于去重和诊断 |
| `timestamp` | Unix 毫秒时间，仅用于日志，不作为安全依据 |

未知非关键字段必须忽略；未知消息类型返回 `unsupported_message`，不能崩溃或继续读取为文件载荷。

## 3. 设备发现

发现包示例：

```json
{
  "type": "discovery_beacon",
  "app": "HMTrans",
  "protocolMin": "0.1.0",
  "protocolMax": "0.2.0",
  "deviceId": "mac-fingerprint-prefix",
  "deviceName": "Linksc 的 MacBook Pro",
  "platform": "macOS",
  "systemVersion": "26.0",
  "ip": "192.168.1.20",
  "port": 51888,
  "receiverState": "ready",
  "pairingRequired": true,
  "capabilities": [
    "resume",
    "folder_semantics",
    "group_transfer",
    "pairing_sas",
    "background_resume"
  ],
  "timestamp": 1783700000000
}
```

`receiverState` 可取 `ready`、`send_only`、`disabled`、`error`。发现包不包含用户文件、历史记录或私密密钥。

接收端优先以 UDP 数据包来源地址作为可连接地址，不能盲信载荷中的 `ip`。广播、单播和网络切换均使用同一结构。

## 4. 能力协商

建立 TCP 后先发送 `hello`：

```json
{
  "type": "hello",
  "app": "HMTrans",
  "protocolVersion": "0.2.0",
  "messageId": "...",
  "timestamp": 1783700000000,
  "deviceId": "mac-fingerprint-prefix",
  "deviceName": "Linksc 的 MacBook Pro",
  "platform": "macOS",
  "systemVersion": "26.0",
  "protocolMin": "0.1.0",
  "protocolMax": "0.2.0",
  "capabilities": ["resume", "folder_semantics", "pairing_sas"]
}
```

接收端返回 `hello_ack`，携带选择的 `selectedProtocol` 和能力交集。没有共同协议时返回错误并关闭连接。

## 5. 配对

首次配对流程：

```text
pairing_start
  → pairing_challenge
  → 双端显示六位短认证码
  → pairing_confirm
  → pairing_result
```

`pairing_start` 和 `pairing_challenge` 携带临时公钥、随机数和设备身份公钥。六位码由双方共同材料派生，不直接在网络中发送。

```json
{
  "type": "pairing_confirm",
  "app": "HMTrans",
  "protocolVersion": "0.2.0",
  "messageId": "...",
  "timestamp": 1783700000000,
  "deviceId": "pad-fingerprint-prefix",
  "accepted": true,
  "identityFingerprint": "sha256:..."
}
```

配对成功只表示设备身份已被用户确认，不代表文件内容已经端到端加密。后续连接需要验证已保存身份，身份变化时重新配对。

## 6. 传输任务模型

群组发送包含一个 `groupId`，每个接收设备拥有独立 `transferId`。一对一发送也可有 `groupId`，但不是必需。

```json
{
  "type": "transfer_offer",
  "app": "HMTrans",
  "protocolVersion": "0.2.0",
  "messageId": "...",
  "timestamp": 1783700000000,
  "transferId": "B96F52A5-8B6E-4B19-9D85-2C12F3F9E13C",
  "groupId": "0F715A7E-C3D9-4A03-8B97-50E526EA4CF1",
  "senderDeviceId": "mac-fingerprint-prefix",
  "receiverDeviceId": "pad-fingerprint-prefix",
  "sourceKind": "folder",
  "payloadKind": "zip",
  "displayName": "项目素材",
  "sourceSize": 2876231120,
  "sourceFileCount": 185,
  "payloadSize": 2511120400,
  "payloadSha256": "012345...",
  "chunkSize": 1048576,
  "resumeSupported": true
}
```

字段说明：

| 字段 | 说明 |
| --- | --- |
| `sourceKind` | `file` 或 `folder`，决定用户语义和最终结果 |
| `payloadKind` | `raw` 或 `zip`，是内部传输载荷 |
| `displayName` | 用户看到的文件或文件夹名，不是内部 ZIP 名 |
| `sourceSize` | 原始来源总大小 |
| `sourceFileCount` | 文件夹内文件数；普通文件为 1 |
| `payloadSize` | TCP 实际发送的字节数 |
| `payloadSha256` | 载荷完整 SHA-256 |

接收端不得使用对方提供的路径，只能使用经过清理的 `displayName` 生成自己的目标路径。

## 7. 接受、拒绝和续传偏移

接受新任务：

```json
{
  "type": "transfer_accept",
  "app": "HMTrans",
  "protocolVersion": "0.2.0",
  "messageId": "...",
  "timestamp": 1783700000000,
  "transferId": "B96F52A5-8B6E-4B19-9D85-2C12F3F9E13C",
  "resumeOffset": 0,
  "lastCompleteChunkSha256": null,
  "reservedName": "项目素材"
}
```

恢复任务：

```json
{
  "type": "transfer_accept",
  "app": "HMTrans",
  "protocolVersion": "0.2.0",
  "messageId": "...",
  "timestamp": 1783700000000,
  "transferId": "B96F52A5-8B6E-4B19-9D85-2C12F3F9E13C",
  "resumeOffset": 536870912,
  "lastCompleteChunkSha256": "abcdef...",
  "reservedName": "项目素材"
}
```

拒绝：

```json
{
  "type": "transfer_reject",
  "app": "HMTrans",
  "protocolVersion": "0.2.0",
  "messageId": "...",
  "timestamp": 1783700000000,
  "transferId": "B96F52A5-8B6E-4B19-9D85-2C12F3F9E13C",
  "reason": "user_rejected",
  "retryable": false
}
```

发送端必须验证 `resumeOffset` 不超过载荷大小、位于合法分块边界，并验证最后完整分块哈希。验证失败时从双方确认的安全偏移恢复，不能直接信任偏移。

## 8. 二进制传输和控制

收到 `transfer_accept` 后，发送端从 `resumeOffset` 开始在客户端到服务端方向发送恰好 `payloadSize - resumeOffset` 字节。

接收端在服务端到客户端方向保持控制消息通道，可在载荷传输期间发送：

- `receiver_pause`：接收端用户暂停，发送端停止写入并关闭数据连接。
- `receiver_cancel`：接收端取消，发送端停止写入；是否删除部分文件由接收端本地策略决定。
- `receiver_status`：低频确认进度或通知等待状态，不用于替代 TCP 流控。

发送端主动暂停时停止写入并正常关闭连接，接收端把未完成任务保存为可恢复。发送端主动取消需要建立短控制连接发送 `transfer_command`：

```json
{
  "type": "transfer_command",
  "app": "HMTrans",
  "protocolVersion": "0.2.0",
  "messageId": "...",
  "timestamp": 1783700000000,
  "transferId": "B96F52A5-8B6E-4B19-9D85-2C12F3F9E13C",
  "command": "cancel",
  "deletePartial": true
}
```

控制消息禁止混入客户端到服务端的原始载荷字节流。

## 9. 完成和失败

接收端写完载荷、落盘并校验后返回：

```json
{
  "type": "transfer_result",
  "app": "HMTrans",
  "protocolVersion": "0.2.0",
  "messageId": "...",
  "timestamp": 1783700000000,
  "transferId": "B96F52A5-8B6E-4B19-9D85-2C12F3F9E13C",
  "success": true,
  "payloadSha256": "012345...",
  "savedName": "项目素材",
  "reason": null,
  "retryable": false
}
```

文件夹只有在校验、解压和最终发布全部成功后才返回成功。仅收到全部 ZIP 字节不算用户任务完成。

## 10. 心跳和超时

- 活动载荷持续流动时不额外发送高频心跳。
- 处于等待用户确认、文件准备或暂时无载荷阶段时，可每 15 秒发送一次 `session_ping`。
- 连续多个周期无响应后进入等待重连，不立即删除部分文件。
- 超时值必须允许大文件校验和文件夹解压阶段显式报告状态，避免误判断连。

## 11. 错误码

| 错误码 | 含义 | 默认是否可重试 |
| --- | --- | --- |
| `discovery_unavailable` | 发现服务不可用 | 是 |
| `protocol_incompatible` | 无共同协议版本 | 否 |
| `pairing_required` | 需要配对 | 是 |
| `identity_changed` | 已配对设备身份变化 | 否，需重新配对 |
| `user_rejected` | 用户拒绝 | 否 |
| `source_unavailable` | 发送来源不可访问 | 是，需重新选择 |
| `destination_unavailable` | 接收目录不可用 | 是 |
| `insufficient_space` | 空间不足 | 是 |
| `connection_lost` | 连接中断 | 是 |
| `resume_mismatch` | 断点或分块哈希不一致 | 是，回退或重传 |
| `hash_mismatch` | 最终 SHA-256 不一致 | 是，从头重传 |
| `archive_invalid` | 文件夹归档无效 | 是 |
| `archive_unsafe_path` | 归档包含不安全路径 | 否 |
| `publish_failed` | 发布到最终目录失败 | 是 |
| `background_interrupted` | 系统中断后台任务 | 是 |
| `cancelled` | 用户取消 | 视是否保留部分文件 |
| `internal_error` | 未分类内部错误 | 视详情 |

用户界面不能只展示错误码，必须同时给出本地化说明和建议操作。

## 12. v0.1 兼容

v0.2 发现包声明 `protocolMin=0.1.0` 和 `protocolMax=0.2.0`。识别到 v0.1 对端时：

- 使用 v0.1 的 `file_meta → receive_decision → 原始字节 → transfer_success/failed` 流程。
- 只支持一对一完整文件发送。
- 不启用配对码、断点续传、文件夹自动还原、群发或后台恢复协商。
- UI 清楚标记“对方为旧版本，正在使用基础传输”。

v0.1 兼容层与 v0.2 引擎分开，不能让旧字段污染 v0.2 数据模型。

### v0.1 兼容线格式

兼容适配器需要识别现有 `file_meta`：

```json
{
  "type": "file_meta",
  "app": "HMTrans",
  "version": "0.1.0",
  "transferId": "B96F52A5-8B6E-4B19-9D85-2C12F3F9E13C",
  "senderDeviceId": "mac-xxxxxx",
  "senderName": "MacBook Pro",
  "senderPlatform": "macOS",
  "fileName": "IMG_0001.DNG",
  "fileSize": 48234432,
  "sha256": "012345...",
  "chunkSize": 1048576,
  "totalChunks": 46
}
```

接收决定：

```json
{
  "type": "receive_decision",
  "accepted": true,
  "reason": null
}
```

传输结果为 `transfer_success` 或 `transfer_failed`，携带 `transferId`、`sha256` 和 `reason`。适配器按 `fileSize` 读取完整原始载荷，不尝试用 v0.2 消息插入旧连接。

## 13. 安全限制

- 所有长度、偏移、名称和 JSON 大小都有上限。
- 文件名去除路径分隔符、控制字符和保留名称。
- ZIP 解压验证每个目标路径位于任务私有目录内。
- 身份私钥和配对密钥不写入普通日志或数据库。
- 诊断日志对路径和网络信息做脱敏。
