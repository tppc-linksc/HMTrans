# v0.3 开发计划

本计划记录 `v0.3.0` 从开发分支 `codex/v0.3.0-screen-cast` 到正式发布的收口过程。双端实现、安全加固、构建、协议测试和当前目标设备核心流程均已完成；2026-07-22 经产品测试确认后，macOS 进入 GitHub Release 发布，HarmonyOS 等待正式 Profile/证书签名和应用市场上传。

## 1. 交付目标

交付可公开测试的 `v0.3.0`：已配对 MatePad 经 HarmonyOS 系统录屏授权后，把硬件编码的 H.264 画面通过加密局域网链路投放到 Mac 独立窗口；支持普通窗口、画中画、全屏、方向变化、停止、有限重连、诊断、深浅色和多窗口形态，同时完整保留 `v0.2.1` 文件互传能力。

首版只做 Pad 到 Mac 的画面，不传声音、不远程控制、不作为副屏、不在 HMTrans 内录制。

## 2. 已完成的工程实现

### HarmonyOS 发送端

- [x] 原生 `OH_AVScreenCapture` 系统屏幕采集和每次会话授权。
- [x] 原生 `OH_AVCodec` H.264 硬件编码、参数集、关键帧与 N-API 二进制回调。
- [x] `PadScreenCastController` 会话状态、设置、停止和页面响应式接入。
- [x] 流畅、均衡、高清三档画质。
- [x] 独立投屏端口、四帧/32 MiB 有界队列、积压丢弃旧帧。
- [x] 2 秒心跳、6 秒超时与 10 秒有限重连。
- [x] 连接页投屏入口、精简状态条、设置页和投屏诊断。
- [x] 仅从 Pad 设备卡片发起投屏，并在 UI 作用域调起 HarmonyOS 系统 Picker。
- [x] 页面隐藏不销毁活动会话，Ability 销毁统一释放定时器、套接字、采集和编码资源。

### HMCS 协议与安全

- [x] 发现广播增加投屏端口、协议版本和 H.264 能力，旧版设备仍可继续传文件。
- [x] 32 字节二进制头、64 KiB 控制上限、8 MiB 媒体上限和严格保留位检查。
- [x] 六位码认证 X25519 临时密钥协商，完成双向证明后双端保存 256 位主密钥；网络消息不携带六位码或长期密钥。
- [x] 文件、解除配对、投屏媒体与网络测试使用 HKDF-SHA256 派生的用途隔离子密钥。
- [x] 接受响应、视频配置、视频帧、心跳、控制和结束载荷使用 AES-256-GCM。
- [x] 每包独立 96 位随机 nonce，固定头作为 AAD，连接内序号严格递增。
- [x] 未配对、旧版无密钥、身份变化、认证失败、重复或乱序消息直接拒绝。

### macOS 接收端

- [x] 独立投屏监听器、单活动会话限制和基于本地配对记录的自动认证。
- [x] `VTDecompressionSession` H.264 解码与 `AVSampleBufferDisplayLayer` 原生显示。
- [x] 独立观看窗口、保持比例、画中画、全屏、统计覆盖层和停止操作。
- [x] 普通窗口与画中画复用同一解码会话，支持四角停靠、位置恢复和越界修正。
- [x] 关闭观看窗口、关闭接收服务、睡眠和应用退出时停止并释放会话。
- [x] 设置页投屏接收开关、独立端口和脱敏诊断。
- [x] Mac 不提供远程启动入口，只接收 Pad 主动建立且通过身份认证的投屏会话。

### 工程质量与回归

- [x] 投屏采集、网络、状态和存储已拆入控制器与服务；`Index.ets` 当前仍超过 1000 行，作为后续结构性重构项保留，不虚报达标。
- [x] 新增实现与关键协议代码使用中文设计注释。
- [x] macOS HMCS 帧头、拆包、加密、未知标志、保留位和超限测试。
- [x] macOS 既有配对、解除配对、并发、断点续传、文件夹和压缩安全测试继续通过。
- [x] HarmonyOS HAP 与 macOS 应用通过本地构建。
- [x] 首次隐私说明与网页隐私政策补充屏幕信息、系统授权、内存显示和无云端边界。

## 3. 统一真机验收与持续回归

当前目标设备核心授权、连接、投屏和停止流程已由产品测试确认通过。以下完整矩阵不能由构建或单元测试替代，未覆盖的长时、极端网络和多设备压力项目继续作为发布后回归，不把它们误写成已经取得性能证据：

- [ ] 首次发起出现 HarmonyOS 系统录屏授权，拒绝后资源完全释放，再次发起可恢复。
- [ ] Pad 发起投屏能调起系统 Picker；拒绝授权、旧版不支持和 Mac 接收服务关闭时显示明确结果。
- [ ] 切换到其他 Pad 应用后能按系统允许范围持续投屏，系统录屏标识和停止入口始终可见。
- [ ] 720p/均衡/高清三档分别验证首帧、可读性、帧率、端到端延迟、CPU、内存和温度。
- [ ] 横竖屏、分屏、浮窗、锁屏、受保护页面和前后台切换行为符合系统规则。
- [ ] Mac 普通窗口、画中画、全屏、缩放、四角停靠、多显示器和系统录屏演示可用。
- [ ] Wi-Fi 短断、切网、Mac 睡眠/唤醒、关闭窗口和关闭接收服务均能明确停止或在 10 秒内恢复。
- [ ] 连续投屏 60 分钟，100 次开始/停止，无崩溃、无残留录屏、无持续增长内存。
- [ ] 投屏与大文件双向传输同时进行，文件 SHA-256、断点和任务状态不受影响，并记录带宽竞争数据。
- [ ] 重新验证文件、文件夹、断点续传、多设备、解除配对、历史、设置和后台传输。
- [ ] v0.2.1 覆盖升级与 v0.3.0 干净安装分别验证配对迁移；旧版无投屏密钥的设备明确要求重新配对。

## 5. 工程约束

- 采集、编码、协议、传输、解码、显示和窗口各自独立，公共接口写中文文档注释。
- 新增实现文件保持在 600 行以内；既有页面或控制器超过 1000 行必须拆分。
- 不引入云 SDK、广告 SDK、账号系统或无必要的大型媒体框架。
- 不在 UI 线程执行视频编码、解码、加密或大缓冲复制。
- 每个活动会话都必须有停止入口，应用生命周期结束时统一清理。
- 文件传输与投屏状态互不复用；任何性能优化都不能牺牲文件字节正确性。

## 6. 后续版本

`v0.3.1` 计划完成两项独立能力：Mac 主动请求 Pad 投屏，以及经 Pad 单独授权的触控、键盘和鼠标反向操作。远程启动只能调起 Pad 系统投屏确认，不能静默录屏；观看权限与控制权限必须分离，任一端停止或 Pad 撤销控制后立即失效。

声音和更强的自适应码率不进入 `v0.3.0`，继续在后续 `v0.3.x` 以独立隐私、权限、协议和验收方案评估。`v0.3.1` 的详细范围见 [09-v0.3.1远程启动与反向控制规划.md](09-v0.3.1远程启动与反向控制规划.md)。

## 7. 完成定义

只有同时满足以下条件，`v0.3.0` 才算完成：

1. 产品、协议、代码、隐私和对外描述一致。
2. 目标双端真机能完成授权、启动、方向变化、窗口模式切换、停止和异常恢复。
3. 文件互传没有回归，投屏与文件任务并存不破坏正确性。
4. 关键性能与稳定性指标有真实设备数据，不用构建成功代替真机验收。
5. 发布包、版本号和官网内容经过独立核对。
