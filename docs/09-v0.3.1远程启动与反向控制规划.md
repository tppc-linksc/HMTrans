# v0.3.1 远程启动与反向控制规划

本文记录 `v0.3.1` 的计划范围。它不属于当前 `v0.3.0` 实现、测试包或发布说明；只有双端代码、平台权限、真机安全和交互验收全部完成后，才能对外声明支持。

## 1. 版本目标

`v0.3.1` 在保持 MatePad 到 Mac 单向画面链路的基础上增加：

1. **Mac 主动请求投屏**：用户在 Mac 选择已配对且在线的 Pad，向 Pad 发送经过配对密钥认证的一次性启动请求。Pad 必须在前台调起 HarmonyOS 系统投屏确认，Mac 不能静默开始录屏。
2. **Mac 反向操作 Pad**：投屏建立后，Pad 用户可再单独允许 Mac 发送触控、鼠标和键盘输入。观看权限不自动包含控制权限。

声音、扩展显示器、云端控制和无人值守远程访问不进入本版本。

## 2. 产品交互

- Mac 主窗口提供轻量入口，选择在线 Pad 后只发送启动请求；入口样式必须重新设计，不复用 `v0.3.0` 已删除的左下角浮动按钮和设备弹窗。
- Pad 收到请求后显示来源设备，并由系统投屏 Picker 完成屏幕采集授权。Pad 应用不可交互、已退出或系统拒绝时，Mac 显示明确失败原因。
- 画面连接成功后，Mac 观看窗口显示“请求控制”。Pad 再显示独立控制授权，允许用户选择只观看或允许控制。
- Pad 持续显示正在投屏/正在受控状态和停止入口；撤销控制不必结束观看，停止投屏必须同时撤销控制。
- 多 Pad 同时投屏时，键盘和鼠标事件只发送到用户当前聚焦的一个窗口，不能广播到全部设备。

## 3. 协议与安全边界

- 远程启动、控制授权和输入事件使用独立消息类型及 HKDF 用途密钥，不能直接复用文件认证或媒体密钥。
- 每个请求包含目标安装身份、会话 ID、连接 ID、时间窗、一次性请求 ID 和严格递增序号，拒绝伪造、过期、乱序和跨连接重放。
- 坐标使用相对画面尺寸的标准化值，并绑定当前方向、分辨率和显示区域；方向变化后旧坐标事件立即失效。
- 文本、按键和触控事件只在内存中转发，不写入历史、诊断或日志；密码框、受保护页面和系统禁止注入的区域按平台规则拒绝。
- 配对、观看、控制是三个不同层级。解除配对立即终止媒体和控制会话，并删除对应本地信任材料。

## 4. 平台与权限门

- 实现前必须以目标 HarmonyOS 6.1/API 23+ 真机和正式应用身份确认可用的官方输入能力、所需 ACL/权限、应用市场开放范围及用户授权方式。
- 若普通第三方应用无法通过公开能力注入通用触控或按键，不使用无障碍滥用、调试接口、系统签名伪装或其他绕过平台安全边界的兼容实现。
- Mac 端只采集当前观看窗口内的鼠标、触控板和键盘事件；窗口失焦、控制开关关闭或会话异常时立即停止发送。

## 5. 完成与验收门

- Mac 请求不能绕过 Pad 系统录屏确认，拒绝、超时、重放和应用不在前台都有明确状态。
- 只观看与允许控制可独立选择，权限提示、状态标识和撤销入口在深浅色、横竖屏、分屏和浮窗下可用。
- 触控坐标、滚动、鼠标按键、常用键盘输入、方向变化和不同缩放比例在真机上正确映射。
- 输入链路断开、窗口失焦、锁屏、受保护页面、解除配对和任一端停止时不残留控制能力。
- 文件传输、双路投屏、网络测试、画中画和 v0.3.0 全量回归继续通过。
- 隐私政策、权限申请、应用市场说明、协议文档、自动化测试和真机证据与实际实现一致。
