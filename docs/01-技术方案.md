# 技术方案

## 总体架构


## 系统适配基线

- HarmonyOS 端以 HarmonyOS 6.1.0 / API 23 及以上为目标，不再按 HarmonyOS 5.0 的最低能力设计交互和能力边界。
- HarmonyOS 端优先使用 ArkTS / ArkUI / HarmonyOS 原生能力，包括窗口化/分屏适配、拖拽、沉浸式界面、系统文件能力、后台任务和新系统视觉特性。
- macOS 端以 macOS 26 及以上为目标，优先使用 SwiftUI、AppKit、菜单栏、Dock 拖拽、系统材质、深浅色模式和新系统原生能力。
- 兼容旧系统不是当前优先级；如果某个新系统 API 会明显改善体验，优先按新系统实现，再在文档中说明最低系统要求。

```text
MacBook Pro  <---- 同一 Wi-Fi 局域网 ---->  MatePad
```

两端都运行 PureSend。文件只在局域网内传输，不经过服务器。

系统分为六层：

1. 设备发现层。
2. 连接层。
3. 协议层。
4. 传输层。
5. 校验层。
6. UI 层。

## v0.1 设备发现

`v0.1` 使用 UDP 广播发现同一局域网内的 PureSend 设备：

- Mac 定时广播自己的设备信息。
- MatePad 定时广播自己的设备信息。
- 收到广播后，如果 `app` 字段是 `PureSend`，就显示到附近设备列表。
- 已连接过的设备会记录地址，广播暂时丢失时可做短时间 fallback 探测。
- 离线设备通过 TTL 清理，避免杀掉一端后另一端长期显示假连接。

广播内容示例：

```json
{
  "app": "PureSend",
  "version": "0.1.0",
  "deviceName": "MacBook Pro",
  "platform": "macOS",
  "ip": "192.168.1.20",
  "port": 51888,
  "deviceId": "mac-xxxxxx"
}
```

## v0.1 文件传输

文件传输使用 TCP。

原因：

- TCP 能保证顺序和可靠传输。
- 大文件传输更简单。
- 两端实现成本比自定义可靠 UDP 低。

流程：

1. 发送端连接接收端 TCP 端口。
2. 发送端发送文件元数据 JSON。
3. 接收端根据设备信任状态决定是否弹确认。
4. 接收端同意后返回确认消息。
5. 发送端按 1MB 分片发送原始字节。
6. 接收端写入 `.part` 临时文件。
7. 接收端计算 SHA-256。
8. 校验一致后改成最终文件名。
9. 接收端返回成功或失败消息。

文件夹传输：

- 发送端先把文件夹压缩为 zip。
- 传输协议仍按单文件处理。
- 接收端保存 zip，不在 `v0.1` 自动解压。


`v0.2` 文件夹语义传输：

- 用户发送的是文件夹，接收端最终必须还原为文件夹。
- 内部可以继续使用 zip / tar / 自定义归档作为传输载体，但归档文件名、解包过程和临时文件不暴露给用户。
- 传输 UI 展示原始文件夹名称、目录大小、文件数量和速度，不展示内部归档包。
- 接收成功后自动解包到目标目录，并清理临时归档。

## 后台保活与后台传输

`v0.1`：

- 双端启动后尽量保持接收服务常驻。
- 应用回到前台时重新启动发现服务并清理过期设备。
- 锁屏、系统省电、进程被杀或网络切换时，传输可能中断；中断后依赖重新发现和重新发送。

`v0.2`：

- 后台保活和后台传输必须作为核心能力处理，传输过程中退到后台或短时间锁屏不应中断。
- HarmonyOS 端需要使用 HarmonyOS 6.1/API 23+ 合适的后台任务、长时任务、通知或网络保活能力，并清楚提示系统授权状态。
- macOS 端需要保证菜单栏常驻、接收服务常驻，窗口关闭不影响传输。
- 如果系统强制暂停网络或杀进程，必须把当前 transferId、offset、校验状态和临时文件写入可恢复状态，重新连接后进入断点续传。
- UI 需要区分“后台传输中”“等待系统恢复”“已暂停，可继续”“已中断，可续传”。

## UI 状态同步

双端 UI 都维护两类列表：

- 当前传输：显示传输方向、进度、实时速度、已传大小/总大小，后续支持暂停、继续、停止。
- 历史记录：显示方向、结果、文件大小、文件格式、平均速度、保存位置或失败原因，后续支持单条删除和清空。

接收端也必须创建当前传输记录，不能只在发送端显示进度。

## 完整性校验

发送前计算源文件 SHA-256。

接收完成后，接收端重新计算保存文件的 SHA-256。

只有两边一致才认为成功。

## 安全边界

`v0.1`：

- 只在可信局域网使用。
- 首次连接或首次接收需要用户确认。
- 已信任设备后续自动连接/接收。
- 默认保存到独立目录。

`v0.2`：

- 加入配对码。
- 加入一对多传输的设备选择和结果隔离。
- 加入断点续传所需的失败文件状态和续传入口。
- 加入后台保活和后台传输保证。
- 设备发现和连接消息携带完整设备名称、平台、系统版本和能力字段。

后续版本可考虑：

- TLS 或自定义加密通道。
- 系统通知。
