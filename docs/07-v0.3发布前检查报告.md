# v0.3 发布前检查报告

> 检查日期：2026-07-18
> 目标版本：v0.3.0
> 分支：`codex/v0.3.0-screen-cast`

## 结论

MatePad 到 Mac 单向投屏的双端实现已经完成，并在 2026-07-18 完成安全与稳定性加固：配对不再传输六位码或长期密钥，文件、解除配对、投屏控制和媒体均使用用途隔离密钥认证；Mac 解码与 Pad UI 回调均使用有界队列。代码继续处于统一真机测试阶段，不是公开发布候选，因为系统 Picker、硬件编码、持续画面、方向变化、后台行为、长时间稳定性和真实端到端性能只能由目标设备验收。

稳定版 `v0.2.1` 的文件互传与公开下载继续维持；v0.3 分支不得修改已发布 Release 附件，也不能在真机验收前把投屏写成稳定版已发布能力。

## 已完成

- [x] 产品边界：MatePad 到 Mac 单向 H.264 画面，不含声音、控制、副屏或内置录制。
- [x] HarmonyOS 原生屏幕采集、硬件 H.264 编码和 N-API 二进制回调。
- [x] HMCS 独立端口、能力发现、帧上限、有界队列、心跳和有限重连。
- [x] 六位码认证 X25519 临时密钥协商，双向证明后派生 256 位主密钥；六位码与主密钥不进入网络 JSON。
- [x] 文件元数据、解除配对和主动投屏控制使用用途隔离 HMAC；媒体与网络测试使用独立派生密钥和 AES-256-GCM。
- [x] Mac 配对身份认证、VideoToolbox 解码、独立窗口、画中画、全屏和停止。
- [x] Mac 主动投屏请求、HMAC-SHA256、30 秒时间窗、防重放和 Pad 系统 Picker 调起链路。
- [x] 双端设置、状态、诊断、隐私说明和错误反馈接入。
- [x] 投屏画面不落盘、不进入数据库、不进入文件历史、不上传云端。
- [x] macOS HMCS、主动请求 HMAC、配对/文件认证与既有文件传输共 24 项 Swift 测试通过。
- [x] HarmonyOS HAP 与 macOS Release App 在本轮变更后重新构建通过。
- [x] 投屏实现按控制器、服务、协议和 Native 模块拆分；`Index.ets` 仍高于 1000 行目标，列为结构性重构项而不虚报达标。
- [x] 开源目录只保留可以随项目公开的产品、技术、协议与验收资料。

- macOS：Swift Testing 共 24 项全部通过，包含 X25519 六位码证明、文件元数据认证、解除配对认证、跨连接媒体防重放、断点续传、并发接收、ZIP 安全和慢速连接隔离。
- HarmonyOS：`src/test` 已补充协议、安全和状态用例；当前工程没有配置 `ohosTest` product，命令行 HAP 构建不会执行这些用例，因此不把 HAP 构建写成 HarmonyOS 测试通过。

## 当前风险

1. HarmonyOS 屏幕采集授权和后台行为以系统策略为准，构建成功不能证明切换其他应用后仍能持续采集。
2. H.264 编码器输出格式、持续 P 帧、方向变化和 60 fps 请求能力必须用真实画面验证，固定协议测试不能代替解码实测。
3. TCP 首版依靠丢弃旧帧和有界队列控制延迟，弱网和大文件并行仍需要真实网络数据。
4. 2026-07-18 前建立的配对记录没有安全版本 1 的主密钥，文件传输和投屏都会明确要求删除设备后重新配对，这是主动阻断不安全兼容路径。
5. macOS 当前没有开发者账号公证能力；测试包可用，但公开分发必须继续如实提示系统安全限制。

## 对外描述规则

- 官网和稳定版 README 继续以 `v0.2.1` 为公开能力基线。
- 开发文档可以写“代码与构建完成、进入真机验收”，不能写“v0.3 已发布”。
- 隐私政策可以提前覆盖开发中的屏幕投放处理边界，但应用市场权限声明必须与最终提交包一致。
- 真机阻塞项全部通过后，再更新本报告、官网、Release Notes 和应用市场文案。
