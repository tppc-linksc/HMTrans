# v0.3 发布前检查报告

> 检查日期：2026-07-22
> 目标版本：v0.3.0
> 分支：`codex/v0.3.0-screen-cast`

## 结论

MatePad 到 Mac 单向投屏的双端实现、安全与稳定性加固、构建、协议测试和当前目标设备核心流程已经完成。2026-07-22 用户确认 v0.3.0 测试通过，因此 macOS 版可作为 GitHub Release 候选发布；HarmonyOS 代码与目标设备流程通过，但正式应用市场包仍须在权限审核完成后使用正式 Profile/证书重新签名、复核并上传，测试 HAP 不能代替商店包。

v0.3.0 继续保留 v0.2.1 文件互传能力，并新增仅从 MatePad 发起的单向投屏。Mac 主动请求投屏和反向操作 Pad 明确留到 v0.3.1，不得在当前版本文案中宣称支持。

## 已完成

- [x] 产品边界：MatePad 到 Mac 单向 H.264 画面，不含声音、控制、副屏或内置录制。

- [x] 文件元数据和解除配对使用用途隔离 HMAC；媒体与网络测试使用独立派生密钥和 AES-256-GCM。
- [x] Mac 配对身份认证、VideoToolbox 解码、独立窗口、画中画、全屏和停止。

- [x] 投屏画面不落盘、不进入数据库、不进入文件历史、不上传云端。
- [x] macOS HMCS、配对/文件认证与既有文件传输测试通过。
- [x] HarmonyOS HAP 与 macOS Release App 在本轮变更后重新构建通过。
- [x] 投屏实现按控制器、服务、协议和 Native 模块拆分；`Index.ets` 仍高于 1000 行目标，列为结构性重构项而不虚报达标。
- [x] 开源目录只保留可以随项目公开的产品、技术、协议与验收资料。

## 自动验证记录

- HarmonyOS：`assembleHap --no-daemon` 成功，ArkTS 与 Native C++ 编译通过。仅有 DevEco 自带打包工具 `fastjson2` 调用已弃用 JDK API 的外部警告。
- macOS：Release App 构建、ad-hoc 签名与 `codesign --verify --deep --strict` 通过；生成的 Info.plist 包含局域网与 Wi-Fi 名称用途说明。
- macOS：Swift Testing 共 22 项全部通过，包含 X25519 六位码证明、文件元数据认证、解除配对认证、跨连接媒体防重放、断点续传、并发接收、ZIP 安全和慢速连接隔离。
- HarmonyOS：`src/test` 已补充协议、安全和状态用例；当前工程没有配置 `ohosTest` product，命令行 HAP 构建不会执行这些用例，因此不把 HAP 构建写成 HarmonyOS 测试通过。
- 2026-07-22：产品测试确认当前目标 MatePad 到 Mac 的 v0.3.0 核心投屏流程通过；此结论不等价于上方仍未完成的长时、极端网络和多设备压力数据。

## 当前风险

1. HarmonyOS 屏幕采集授权和后台行为以系统策略为准，构建成功不能证明切换其他应用后仍能持续采集。
2. H.264 编码器输出格式、持续 P 帧、方向变化和 60 fps 请求能力必须用真实画面验证，固定协议测试不能代替解码实测。
3. TCP 首版依靠丢弃旧帧和有界队列控制延迟，弱网和大文件并行仍需要真实网络数据。
4. 2026-07-18 前建立的配对记录没有安全版本 1 的主密钥，文件传输和投屏都会明确要求删除设备后重新配对，这是主动阻断不安全兼容路径。
5. macOS 当前没有开发者账号公证能力；测试包可用，但公开分发必须继续如实提示系统安全限制。

## 对外描述规则

- macOS 官网、README 和 GitHub Release 可以把 `v0.3.0` 作为稳定版，并如实说明 ad-hoc 签名和未公证限制。
- HarmonyOS 只能写“v0.3.0 已完成目标设备测试，待正式签名、上传和审核”；应用市场显示上架前不得写成已发布。
- 隐私政策和应用市场权限声明必须与最终提交包一致；GitHub Release 不提供测试 HAP 作为应用市场包的替代品。
- 长时、弱网、双路和大文件并行等尚未取得完整数据的场景继续写为回归项目，不把单次成功扩张为全场景性能结论。
