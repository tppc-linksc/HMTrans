# v0.3 投屏协议

本文定义 HMTrans Cast Stream（简称 `HMCS`）`0.3.0`。它只用于实时屏幕画面和会话控制，与现有文件传输协议完全独立。

## 1. 服务发现扩展

设备发现广播在保持旧字段兼容的基础上增加可选字段：

```json
{
  "screenCastPort": 51890,
  "screenCastProtocolVersion": "0.3.0",
  "screenCastCapabilities": ["receive-h264"]
}
```

- Pad 发送 `send-h264` 和 `remote-start-picker`，Mac 发送 `receive-h264`。
- 旧版本忽略未知字段；没有 `screenCastPort` 的设备仍可传文件，但不显示投屏操作。
- 发现广播只说明能力，不能代表配对、授权或投屏会话已经建立。

## 2. 连接前提

1. 双方已通过文件传输链路的六位码完成配对。
2. 当前发现身份与已保存安装身份一致。
3. 用户可在 Pad 直接开始，也可由 Mac 通过文件控制端口发送经过配对密钥认证的启动请求。
4. Pad 取得 HarmonyOS 系统屏幕采集授权；Mac 请求不能代替或绕过系统确认。
5. Pad 主动连接 Mac 投屏端口，Mac 校验安装身份、指纹和 256 位配对共享密钥并完成 HMCS 握手。

任一条件不满足都不得开始发送视频帧。

### 2.1 Mac 主动投屏请求

Mac 复用已配对设备的文件控制端口发送一行 JSON；这只是拉起 Pad 系统确认的控制请求，不属于 HMCS 媒体连接：

```json
{
  "type": "screen_cast_start_request",
  "app": "HMTrans",
  "version": "0.2.0",
  "requesterDeviceId": "mac-install-id",
  "requesterFingerprint": "mac-install-fingerprint",
  "targetDeviceId": "pad-install-id",
  "requestId": "uuid",
  "issuedAt": 1721217600000,
  "signature": "hmac-sha256-hex"
}
```

签名规范文本按上述字段顺序用 `|` 连接，使用配对时保存的 256 位共享密钥执行 HMAC-SHA256。Pad 必须校验当前双向在线状态、安装身份、目标设备、30 秒时间窗和未使用过的 `requestId`，先返回对应 `screen_cast_start_response`，再在 UI 作用域调起系统 Picker。请求失败、应用不在可交互状态或系统拒绝时不得启动采集。

## 3. 二进制帧

所有多字节整数使用网络字节序。固定头长度为 32 字节：

| 偏移 | 长度 | 字段 | 说明 |
| --- | --- | --- | --- |
| 0 | 4 | magic | ASCII `HMCS` |
| 4 | 1 | major | 主版本，首版为 `0` |
| 5 | 1 | minor | 次版本，首版为 `3` |
| 6 | 1 | type | 消息类型 |
| 7 | 1 | flags | 消息标志 |
| 8 | 2 | headerLength | 固定为 `32` |
| 10 | 2 | reserved | 必须为 `0` |
| 12 | 4 | payloadLength | 载荷字节数 |
| 16 | 8 | sequence | 当前连接内单调递增序号 |
| 24 | 8 | ptsUs | 单调时钟微秒；无时间戳消息填 `0` |

接收端必须先验证 magic、版本、头长度和载荷上限，再分配载荷缓冲区。单帧载荷上限为 8 MiB，控制 JSON 上限为 64 KiB。

## 4. 消息类型

| type | 名称 | 载荷 |
| --- | --- | --- |
| `0x01` | `HELLO` | UTF-8 JSON |
| `0x02` | `HELLO_ACK` | UTF-8 JSON |
| `0x03` | `VIDEO_CONFIG` | UTF-8 JSON |
| `0x04` | `VIDEO_FRAME` | H.264 Annex B |
| `0x05` | `STREAM_CONTROL` | UTF-8 JSON |
| `0x06` | `HEARTBEAT` | UTF-8 JSON 或空 |
| `0x07` | `ERROR` | UTF-8 JSON |
| `0x08` | `END` | UTF-8 JSON |

未知类型、未知主版本、非零保留字段或未知标志位都按无效帧处理并关闭连接，避免双端对敏感媒体数据产生不同解释。

## 5. 标志位

| 位 | 名称 | 含义 |
| --- | --- | --- |
| `0x01` | `ENCRYPTED` | 载荷使用配对共享密钥进行 AES-256-GCM 加密 |
| `0x02` | `KEY_FRAME` | `VIDEO_FRAME` 是 IDR/关键帧 |
| `0x04` | `CODEC_CONFIG` | `VIDEO_FRAME` 包含 H.264 参数集数据 |

其余保留位必须发送为 `0`。除首个 `HELLO` 和拒绝响应外，接受响应、媒体与控制消息必须设置并实际执行 `ENCRYPTED`，不能只设置标志。

## 6. 握手

### HELLO

```json
{
  "app": "HMTrans",
  "protocol": "0.3.0",
  "deviceId": "sender-install-id",
  "deviceName": "MatePad Pro",
  "identityFingerprint": "sender-install-fingerprint",
  "sessionId": "uuid",
  "codec": "h264-annexb",
  "width": 1920,
  "height": 1200,
  "frameRate": 30
}
```

### HELLO_ACK

```json
{
  "accepted": true,
  "sessionId": "uuid",
  "reason": null
}
```

Mac 用 `deviceId` 在本地配对记录中查找共享密钥，并同时校验身份指纹。接受响应必须使用该密钥加密；明文 ACK 只允许表达拒绝。Pad 无法用本地共享密钥解密接受响应时必须终止会话。`sessionId` 每次发起随机生成，连接内序号严格递增，重复或乱序的已认证消息视为无效。

## 7. 视频配置与帧

`VIDEO_CONFIG` 示例：

```json
{
  "codec": "h264-annexb",
  "width": 1920,
  "height": 1200,
  "frameRate": 30
}
```

- 视频帧使用 Annex B NALU，起始码必须保留。
- 第一帧前必须发送 `VIDEO_CONFIG`，并紧跟关键帧。
- 参数集以带 `CODEC_CONFIG` 标志的视频帧发送；启动和重连都必须先恢复参数集，再发送关键帧。
- 方向、分辨率或帧率变化时重新发送 `VIDEO_CONFIG`，随后发送对应参数集和关键帧。
- PTS 必须单调不减；异常回退时间戳的帧应丢弃。

## 8. 控制消息

`STREAM_CONTROL` 支持：

```json
{ "command": "stop" }
```

- HMCS 会话内 Mac 到 Pad 的控制命令只有 `stop`；开始投屏请求使用 2.1 节的已配对控制端口，不与媒体帧混发。断线重连由 Pad 本地重建编码边界并等待参数集和关键帧。
- 首版没有远程输入、远程调码率、音量、麦克风、录制或剪贴板控制消息。

## 9. 心跳与重连

- 双方每 2 秒发送心跳，心跳可携带最后接收序号、队列长度和简单网络统计。
- 6 秒没有收到任何有效消息时进入重连状态。
- 10 秒重连窗口内最多进行有限次数重试，采用短退避，不无限循环。
- 新连接使用同一 `sessionId` 和新的序号空间，重新执行基于配对共享密钥的接受响应认证。
- 重连成功后 Pad 重新发送视频配置，并从参数集和关键帧开始恢复；Mac 清空旧解码状态。
- 超过窗口后发送/记录结束原因，用户需重新发起投屏和系统授权。

## 10. 结束与错误

`END` 必须包含结束原因：

```json
{ "reason": "用户已停止投屏" }
```

基础错误码：

| 错误码 | 含义 |
| --- | --- |
| `CAST-AUTH-001` | 未配对或身份不匹配 |
| `CAST-AUTH-002` | 配对共享密钥缺失或认证响应无效 |
| `CAST-PROTO-001` | 协议版本不兼容 |
| `CAST-PROTO-002` | 帧头、长度或消息顺序无效 |
| `CAST-BUSY-001` | 接收端已有投屏会话 |
| `CAST-CAPTURE-001` | 系统屏幕采集未授权或被停止 |
| `CAST-CODEC-001` | 编码器或解码器无法启动 |
| `CAST-NET-001` | 网络断开且重连失败 |
| `CAST-SEC-001` | 加密协商或完整性校验失败 |

错误消息不得包含配对码、令牌、密钥、完整 IP、完整本地路径或屏幕内容。

## 11. 加密与完整性

- 配对成功时双方生成并保存 256 位随机共享密钥；六位码只用于人工首次确认，不直接作为媒体加密密钥。
- 除首个 `HELLO` 与明文拒绝外，载荷使用 AES-256-GCM 加密并校验完整性。每包生成独立 96 位随机 nonce，32 字节 HMCS 固定头作为 AAD，密文格式为 nonce、ciphertext、16 字节认证标签。
- 连接内序号必须严格递增；接收端在解密前拒绝重复或乱序消息，避免同一连接内的包重放。
- 不存在可退回的明文媒体模式；无法找到配对密钥或认证失败时直接关闭连接。

## 12. 兼容规则

- 文件传输 `0.2.0` 与 HMCS `0.3.0` 分别协商，互不推导。
- 主版本不一致直接拒绝；相同主版本的较新次版本必须能忽略未知可选字段。
- 发送端使用本次 `HELLO` 声明的编码参数；Mac 只能接受 `h264-annexb` 和安全范围内的尺寸、帧率。
- 协议字段变化必须先修改本文，再修改双端模型、解析器和测试。
