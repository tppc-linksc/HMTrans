# HarmonyOS v0.3 投屏实现说明

本文记录 HarmonyOS 端已经接入的 `v0.3.0` 投屏实现、Mac 双路接收和真机测试边界。当前目标 MatePad 与 Mac 的核心流程已经通过产品测试；HarmonyOS 应用市场包仍需使用审核通过的正式 Profile/证书重新签名、复核并上传，测试 HAP 不能代替商店包。

## 1. 平台边界

- HarmonyOS 6.1.0 / API 23+，优先面向 MatePad。
- Pad 只作为画面发送端；Mac 作为接收和显示端。
- 用户只从 Pad 的已连接 Mac 设备卡片发起；每次会话都必须由 HarmonyOS 系统屏幕采集界面明确授权。Mac 主动请求入口留到 v0.3.1。
- 首版只编码 H.264 画面，不采集音频、不接收远程输入、不充当副屏。

采集与编码在 Native 模块完成，HMCS 网络与加密在 ArkTS 服务完成，会话状态在独立控制器完成；`Index.ets` 只保存页面状态并转发生命周期，不直接持有编码器或套接字。

## 3. ArkTS 与 Native 边界

ArkTS 负责：

- 设备能力判断、会话状态、页面操作和设置持久化。
- HMCS 握手、AES-256-GCM、心跳、四帧有界队列、有限重连和停止通知。
- 系统录屏结果、编码错误和网络错误映射为用户可读状态。

Native 负责：

- `OH_AVScreenCapture` 系统屏幕采集会话。
- `OH_AVCodec` H.264 硬件编码、参数集、关键帧和方向相关输出。
- 把编码后的二进制数据直接回调给 ArkTS，不转换成 Base64 或 JSON。

N-API 暴露 `startCapture/stopCapture/requestKeyFrame/isCapturing/setFrameCallback/setStateCallback` 等最小接口。回调只携带编码数据、单调时间戳和系统状态，不持有页面对象。

## 4. 实际启动顺序

1. 用户在 Pad 点按已连接 Mac 的“投屏”。
2. 控制器读取配对共享密钥和对方广播的投屏端口。
3. Pad 建立 HMCS 连接并发送明文 `HELLO`。
4. Mac 校验安装身份、指纹和共享密钥，返回 AES-256-GCM 加密的接受 ACK；拒绝可用明文原因返回。
5. 认证通过后调用 Native 采集，HarmonyOS 显示系统屏幕采集授权。
6. 授权成功后发送加密视频配置、参数集和关键帧，进入 `casting`。
7. 活动会话启动心跳；页面隐藏不会销毁会话，Ability 销毁会统一清理。

顺序不会调整为“先采集再连接 Mac”，避免无接收目标时提前采集屏幕。

## 5. 状态与生命周期

- 页面状态使用 `idle/connecting/authorizing/casting/reconnecting/failed`。
- `start`、`stop` 和系统回调幂等；切换设备前会先停止旧会话。
- 系统停止录屏、Native 错误、身份失败或 10 秒重连失败时结束会话并释放资源。
- 停止后清理采集器、编码器、帧队列、socket、心跳定时器和回调引用。
- 当前系统数据传输后台能力继续服务活动文件任务；投屏能否在用户离开 HMTrans 后保持采集必须以目标设备系统行为验收。

## 6. 性能约束

- 三档参数：流畅 `1280×840/30 fps/5 Mbps`，均衡 `1920×1260/60 fps/14 Mbps`，高清 `2560×1680/60 fps/24 Mbps`。60 fps 是采集请求上限，实际值必须按设备输出如实显示。
- 待发送视频最多 4 帧或 32 MiB；积压时移除最旧帧并记录丢帧数。
- 首次启动和重连都等待参数集与关键帧，不把不可解码的预测帧发给 Mac。
- 视频编码在系统硬件编码器中运行；ArkUI 只按低频统计刷新，不随每帧重建页面。
- 单帧上限 8 MiB，控制消息上限 64 KiB，避免错误对端触发无界内存。

## 7. UI 接入

- 继续使用连接 / 历史 / 设置三个原生 Dock 入口。
- 仅已连接且明确广播兼容投屏能力的 Mac 显示“投屏”按钮。
- 活动会话使用精简状态条，显示目标、阶段、帧率、码率、丢帧和停止操作。
- 设置页独立选择 720P/1080P/2K 级与 30P/60P，并按真机硬件编码能力禁用不支持组合。
- 设置页提供 Pad→Mac 投屏网络测试，输出吞吐、平均延迟、抖动和推荐档位。
- Mac 最多同时接收两路，双路准入限制为 1080P 级 / 30P，每路独立窗口和停止操作。
- 横竖屏、分屏、浮窗、深色模式和系统手势区域仍需真机逐页确认。

## 8. 隐私与诊断

- 首次隐私说明明确屏幕可能包含通知和个人信息，但不能替代每次系统录屏授权。
- Phone/Tablet 使用 `OH_AVScreenCapture_StrategyForPickerPopUp(..., true)` 显式拉起系统 Picker；
  `ohos.permission.CAPTURE_SCREEN` 属于系统级权限，普通应用不在清单中申请，也不以运行时权限替代 Picker。
- `ohos.permission.CUSTOM_SCREEN_RECORDING` 在当前 SDK 中仅面向 2in1 且采用系统设置授权，不适用于 MatePad；`CAPTURE_SCREEN`、`EXEMPT_CAPTURE_SCREEN_AUTHORIZE` 和 `CAPTURE_SCREEN_ALL` 均为系统核心权限。应用市场普通 Pad 包不能实现无确认远程录屏。
- 画面通过已配对共享密钥加密，只在 Mac 内存中解码显示；不保存截图、视频帧或解码文件。
- 历史页不增加投屏记录；诊断只记录阶段、错误、尺寸、帧率、码率、队列和重连次数。
- 密钥、配对码、身份原值、完整 IP、完整路径和帧内容不得进入日志。

## 9. 构建与测试边界

当前 HAP 已通过命令行构建，Mac 端测试已验证固定头、拆包、加密、上限和非法标志，目标设备核心投屏流程已由产品测试确认通过。以下项目仍需在正式签名包或持续回归中补充完整证据：

- 系统录屏授权、拒绝恢复和切换其他应用。
- Pad 发起后系统 Picker 正确出现，拒绝授权、Mac 接收关闭和旧版设备路径均有明确结果。
- 三档硬件编码、真实 SPS/PPS、关键帧和方向变化。
- 连续 60 分钟、100 次启停、断网重连、锁屏和资源释放。
- 与大文件传输并行时的正确性、延迟、内存、CPU、温度和能耗。
- 深浅色、横竖屏、分屏、浮窗和系统手势区域。
