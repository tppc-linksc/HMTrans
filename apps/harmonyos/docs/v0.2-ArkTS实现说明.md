# v0.2 ArkTS 实现说明

本文只记录 HarmonyOS 端代码入口。产品、协议与验收基线以仓库根目录 [docs](../../../docs/README.md) 为准。

## 1. 当前结构

```text
entry/src/main/ets/
├── pages/                  三页根状态与 ArkUI 组合
├── components/             连接、历史、设置、弹窗和原生 Dock
├── controllers/            网络/配对控制器、发送与任务控制器
├── services/               发现、TCP、后台、归档、目录、诊断和偏好
├── persistence/            relationalStore、schema 与迁移
├── models/                 可持久化任务模型
├── common/                 实际 0.2 协议、格式化与错误文本
└── entryability/           窗口、白色启动和页面加载
```

`Index.ets` 只保留响应式页面状态、生命周期和 UI 事件桥接；socket、归档、数据库和后台任务由独立对象持有。网络/配对/在线状态由 `PadNetworkController` 协调，多目标发送和暂停/恢复/取消由 `PadTransferController` 协调。

## 2. 启动顺序

1. 读取本地隐私版本；未同意时只显示应用自己的隐私弹窗。
2. 同意后加载设置、安装身份、信任关系、数据库和未完成任务。
3. 独立启动 UDP 发现并读取当前 Wi-Fi 名称。
4. 检查或申请公开下载目录；目录可用时启动 TCP 接收器。
5. 下载目录被拒绝时仍保留发现和向外发送能力。
6. 未完成发送从私有 Staging 恢复为暂停或等待重连。

## 3. 存储与权限

- 发送源、未完成接收和内部文件夹归档只放在 `filesDir/HMTrans`，不使用可能被系统主动清理的 `cacheDir`。
- 用户只在 `Download/HMTrans` 看到校验并发布成功的最终文件或文件夹。
- 内部归档和单个 `.part` 不暴露给用户。
- 系统 Picker 授权可能在进程重启后失效，因此发送前复制到持久 Staging。
- “下载目录 → 检查”会验证目录并在必要时重新发起系统授权，结果通过 Toast 返回。

## 4. 传输、后台与性能

- 控制帧为最大 64 KiB 的换行 JSON，载荷使用原始 TCP 字节流；协议见 [v0.2 传输协议](../../../docs/03-v0.2传输协议.md)。
- 文件使用 1 MiB 分块、确认偏移和 SHA-256；每个目标拥有独立任务 ID、进度和恢复点。
- Pad 使用进程级并发门，同时最多运行 2 个发送作业；即使多个任务组并行创建也不会突破上限。
- 仅活动传输期间申请 DATA_TRANSFER 后台任务并展示控制通知；空闲立即释放。
- Picker 大文件复制、目录统计和 SHA-256 在 taskpool 执行。
- 接收连接数限制为 16，单连接 60 秒无有效进度会退出；ZIP 解压前检查路径、符号链接、中央目录边界、条目数、展开大小和压缩比。
- 任务按 `transfer_records` 行级增量持久化；旧单行快照只用于一次迁移，迁移后立即清除。

## 5. 开发约束

- HarmonyOS 目标为 6.1.0 / API 23 及以上，优先使用 ArkTS/ArkUI 6.1 能力。
- 续传、状态机、配对、ZIP 防护和后台引用计数必须保留解释“为什么”的注释。
- 协议字段变化先更新 [协议文档](../../../docs/03-v0.2传输协议.md)。
- 构建通过不能替代应用市场正式包、后台、大文件、三设备和窗口形态真机验收。
- v0.2 不引入云端、账号、中继或自动上传诊断。
