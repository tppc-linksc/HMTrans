# ArkTS 实现说明

HarmonyOS 端需要实现两类能力：

1. 作为发送端，把 MatePad 文件发给 Mac
2. 作为接收端，接收 Mac 发来的文件

## 接收端流程

1. 打开 TCP 监听端口，例如 `51888`
2. 接收连接
3. 读取第一行 JSON
4. 解析为 `FileMeta`
5. 弹窗显示文件名、大小、来源设备
6. 用户同意后发送 `receive_decision`
7. 读取 exactly `fileSize` 字节
8. 写入临时文件
9. 计算 SHA-256
10. 校验成功后保存
11. 返回 `transfer_success`

## 发送端流程

1. 用户通过系统 Picker 选择文件
2. 读取文件名、大小
3. 计算 SHA-256
4. 连接目标 Mac IP 和端口
5. 发送 `file_meta`
6. 等待接收方确认
7. 对方同意后按 1MB 分片发送文件
8. 等待 `transfer_success`

## 权限注意

HarmonyOS 不应该默认读取全盘文件。

发送文件时应通过系统 Picker 获取用户主动选择的文件。

接收文件时第一版优先保存到应用可写目录，后续再支持用户选择保存目录。

## 与 Mac 端保持一致

这些字段必须和 Mac 端完全一致：

- `type`
- `app`
- `version`
- `transferId`
- `fileName`
- `fileSize`
- `sha256`
- `chunkSize`
- `totalChunks`

控制消息必须是 UTF-8 JSON 行，并以 `\n` 结尾。

## 后台传输要求

`v0.1` 先保证应用前台和短时间切后台后的基础接收/发送流程可用，记录锁屏、省电和进程被杀时的限制。

`v0.2` 必须把后台保活和后台传输作为核心能力实现：

- 传输过程中退到后台或短时间锁屏不应直接中断传输。
- 使用 HarmonyOS 6.1/API 23+ 合适的后台任务、长时任务、通知或网络保活能力。
- 系统强制暂停或杀进程时，保存 transferId、offset、临时文件路径和校验状态。
- 重新连接后配合断点续传恢复，不让用户只能重新选择文件。
